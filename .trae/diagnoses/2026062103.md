**S-01 [CMakeLists.txt:19-22] `GLOB_RECURSE` 用于源文件收集 — 构建系统的定时炸弹**

```cmake
file(GLOB_RECURSE SOURCES
    "src/*.cpp"
    "src/*.h"
)
```

→ CMake 官方文档明确警告：`GLOB_RECURSE` 不会在新增/删除文件时自动触发重新配置。开发者添加了新 `.cpp` 文件后必须手动清除 CMake 缓存，否则编译会"无声地"遗漏新模块，产生难以定位的链接错误。

**修复方案：** 显式列举源文件，或使用 `CONFIGURE_DEPENDS`（CMake 3.12+）：

```cmake
file(GLOB_RECURSE SOURCES CONFIGURE_DEPENDS
    "src/*.cpp"
    "src/*.h"
)
```

**S-06 [AuthService.cpp:144-145] 用户密码哈希被加载到内存中的 `SysUser` 结构体**

```cpp
m_currentUser.password = q.value("password").toString();
// ...
m_currentUser.password.clear(); // 第 193 行才清除
```

虽然第 193 行做了清除，但在第 145~193 行之间（约 50 行代码），密码哈希以明文形式驻留在堆内存中。如果此期间发生异常或提前返回，密码哈希将泄漏到 `SysUser` 生命周期中。

→ 应在读取后立即验证并清除，或使用局部变量：

```cpp
QString storedPw = q.value("password").toString();
// ... 验证逻辑 ...
// 不写入 m_currentUser.password
```

实际上，`loadCurrentUser` 的调用者 `login()` 已经单独做了密码验证，`loadCurrentUser` 无需再次读取密码字段。直接从 SELECT 中排除 `password` 列即可。

**S-07 [Utils.h:65-89] 编号生成函数过度设计且存在碰撞风险**

```cpp
inline QString generateOrderNo() {
    QString prefix = "WO";
    QString timestamp = QDateTime::currentDateTime().toString("yyyyMMddHHmmsszzz");
    int rand = QRandomGenerator::global()->bounded(100000, 1000000);
    int seq = nextSequenceNumber();
    return prefix + timestamp + QString::number(rand) + QString("%1").arg(seq, 4, 10, QChar('0'));
}
```

生成的编号长达 35+ 字符（`WO2026062114302512345678901234`），对于 SQLite 单机应用来说完全过度。6位随机数 + 4位自增序列号 + 17位时间戳 = 三重保险，但：
- `QAtomicInt` 序列号在进程重启后归零
- 同一毫秒 + 同一随机数 + 同一序列号 = 理论上可能碰撞（虽然概率极低）

→ 简化为 `WO` + `yyyyMMdd` + 6位自增序列号即可（SQLite 有 UNIQUE 约束保底）。如果需要全局唯一，使用 `QUuid`。

**S-09 [AuthHelpers.cpp:121-125] 使用位置参数 `?` 而非命名参数 `:name` 进行绑定**

**I-05 [DemoDataService.h: 27 个静态初始化函数] 演示数据初始化缺乏事务保护**

27 个 `init*()` 函数依次执行，每个内部执行多条 INSERT。如果在第 15 个函数处失败，前 14 个已写入但后续为空，数据库处于不一致状态。

**I-06 [数据库 schema] 外键约束完全缺失**

1019 行的 schema.sql 中没有一个 `FOREIGN KEY` 约束。`cm_building.estate_id`、`wo_work_order.reporter_id`、`ge_event.grid_id` 等大量关联字段仅靠应用层维护引用完整性。

→ SQLite 支持 `PRAGMA foreign_keys = ON`，建议在 `DatabaseManager::initialize()` 中启用，并为关键关联添加外键约束。对于桌面单机应用来说，这是防止脏数据最简单有效的手段。

**I-07 [数据库 schema] `del_flag` 设计一致但查询中容易遗忘**

所有主要表都有 `del_flag` 字段实现软删除，但 `DatabaseManager::count()` 自动加 `WHERE del_flag = 0`，而直接使用 `QSqlQuery` 的地方（如 `performGlobalSearch`、`handleAction` 中的查询）则容易遗漏此条件。

→ 建议在 `DatabaseManager` 中提供更丰富的查询构建器，自动注入 `del_flag` 过滤。

#### 亮点（值得肯定）

1. **DatabaseManager 的安全设计**：`isKnownTable` 白名单 + `isValidIdentifier` 标识符校验 + `SqlCondition` 结构化条件，三层防护杜绝 SQL 注入，这在桌面应用中属于高水准的安全意识。

2. **密码存储采用 PBKDF2-SHA512**：10000 轮迭代 + 16 字节随机盐，并保留了对旧版 SHA256 格式的向后兼容。这在桌面应用中超出平均水平。

3. **BasePage 信号解耦**：页面通过 `sendNotificationRequested` / `navigateToRequested` 信号与 MainWindow 通信，不直接依赖 MainWindow 接口，是正确的依赖倒置实践。

4. **pageRegistry 统一注册表**：页面创建、面包屑、导航均从单一数据源派生，消除了多处维护的问题，这是本项目中最好的架构决策之一。

5. **UI 设计体系的文档化**：`style.qss` 头部的 Design Tokens 文档（颜色、字体、间距规范）体现了专业的设计思维，「现代档案室」的设计语言一致且有辨识度。

| 考核管理 | P1 | 已实现 | 有指标配置和结果记录，但缺少自动计算得分和排名可视化 |
| 批量数据导入 | P1 | 缺失 | 对于已有的社区，手工逐条录入数千户居民数据是不可接受的 |

#### 偏离预警 🚨

**偏离 1：技术驱动的"功能堆砌"倾向**
→ 当前有 30+ 个子页面、40+ 张数据库表，覆盖面极广。但多个核心流程（如工单闭环）的"最后一公里"体验尚未打磨完善（如操作列不可直观点击、无批量操作）。
→ **影响**：功能广而不深，演示时 impressive，实际使用时处处碰壁。
→ **纠偏建议**：暂停新增功能模块，聚焦打磨工单和事件两条核心流程线的用户体验。

**偏离 2：桌面端定位与用户场景的矛盾**
→ 智慧社区的核心用户（居民、物业维修员）的主要使用场景是移动端。一个纯桌面应用无法满足居民"随手报修"、维修员"现场签到"的需求。
→ **影响**：居民端功能在桌面应用中注定使用率极低。
→ **纠偏建议**：明确本系统定位为"管理后台"，居民端功能（报修、投诉、投票）应标记为"未来需移动端实现"，当前阶段仅保留管理视角的功能。

**偏离 3：SQLite 单机架构与多角色协同的矛盾**
→ 系统设计为多角色协同平台，但 SQLite 是单进程数据库，不支持并发写入。如果多个用户同时使用（即使通过文件共享），数据一致性无法保证。
→ **影响**：仅适用于单机演示场景，无法部署为真正的多用户系统。
→ **纠偏建议**：如果定位为演示/实训项目（版权信息中提到"课程实训项目"），SQLite 可接受。否则需要迁移到 PostgreSQL/MySQL。

#### 模拟需求变更

**新需求 1："居民可以通过微信扫码报修，不需要登录系统"**
→ **合理性**：高。这是实际智慧社区平台的标配。
→ **架构承接能力**：当前架构无法承接。需要：API 层（HTTP Server）+ 微信小程序/H5 前端 + 微信 OAuth 集成。现有的纯桌面架构需要根本性重构为 C/S 或 B/S 架构。

**新需求 2："物业经理需要在手机上查看所有待处理工单并一键催办"**
→ **合理性**：高。
→ **架构承接能力**：同上，需要移动端适配。当前 `QTableWidget` 的桌面 UI 无法迁移到移动端。

**新需求 3："街道办需要每月自动生成社区治理考核报告并导出 PDF"**
→ **合理性**：高。
→ **架构承接能力**：中等。考核数据和指标已存在（`kf_assessment_*` 表），需要添加：自动计算逻辑 + PDF 生成库（如 QPrinter 或第三方库）。可以在当前架构内实现。

#### 战略建议

- **项目应聚焦于**：打磨工单和事件两条核心流程的"最后一公里"体验，确保从创建到关闭的每一步都流畅、直观、可操作
- **建议暂缓**：智能问答（AI 功能在没有可靠后端的情况下只是噱头）、统计分析（在数据量不足的演示环境中无实际价值）
- **建议砍掉**：当前居民端的所有功能入口（报修、投诉、投票等），改为在管理端以"模拟数据"展示，明确标注"居民端需移动端实现"

### 🎨 视觉审查报告

#### 整体印象

- **一句话总评**：「现代档案室」的设计语言有明确的设计主张和高度一致的视觉体系，但在执行层面被 C++ 硬编码样式的碎片化所侵蚀，最终呈现出"设计稿很美，实现打七折"的效果。
- **视觉评分：7 / 10**

#### 严重视觉问题（让我想关掉页面）

**V-01 操作列的"文字伪装" — 用户找不到操作入口**

表格操作列使用 `createActionItem()` 返回纯文本 `QTableWidgetItem`，样式仅靠 `QColor` 区分。"受理"、"派单"这些文字看起来和普通数据单元格没有本质区别。用户不会意识到这里可以点击。

→ **修复方向**：操作列应使用 `createActionCell()` 渲染真正的按钮（项目已有此函数），而非文本项。按钮应有 hover 态和边框，明确传达"可交互"信号。

**V-02 侧边栏图标无文字标签 — 认知负荷过高**

Icon Rail 宽度仅 64px，只有 22x22 的小图标 + tooltip。7 个模块的图标需要用户逐个悬停才能理解含义。对于不常用系统的用户，这是每次数秒的认知负担。

→ 建议将 Icon Rail 扩展为 80px 宽，图标下方添加 10px 的模块名缩写文字（如"档案"、"小区"、"治理"）。

**V-03 `applyCardShadow()` 是空函数 — 卡片完全没有阴影**

```cpp
void applyCardShadow(QWidget* card) {
    card->setGraphicsEffect(nullptr);  // 移除了所有效果
}
```

设计文档中提到 "统计卡片左侧40x40渐变图标区"，但实际的卡片阴影被完全禁用了。注释说是为了避免 Qt6 的 QPainter 冲突，但也没有提供替代方案。

→ 使用多层 `border` 或 `box-shadow` 的 QSS 模拟（虽然 Qt QSS 不原生支持 `box-shadow`，但可以通过嵌套 `QFrame` + 渐变背景模拟层次感）。

#### 视觉改进清单

**V-04 [登录页] 左侧品牌区建筑图形过于抽象**

几何线条建筑感的方向是对的，但 alpha 40 的线条在常规显示器上几乎不可见，建筑块的 `QColor(180, 83, 9, 15)` 填充更是形同虚设。品牌区应该是视觉冲击力最强的区域，目前却需要仔细辨认才能看到内容。

→ 将建筑线条 alpha 提升到 80-120，建筑块 alpha 提升到 40-60。

**V-05 [全局] border-radius 使用不统一**

设计文档声明"全局 2px 圆角"，但实际代码中：
- 登录按钮：4px
- 工具栏：6px
- 搜索框下拉：0px
- 表格：0px
- 卡片：0px

→ 统一为设计文档中声明的值。约束性表面（卡片、对话框）使用 4px，表格、按钮使用 2px 或 0px。

**V-06 [全局] 暖灰色板在低色温显示器上偏黄**

`#FAF9F6`（页面背景）、`#F5F2EB`（交替行）、`#D4D0C8`（分割线）在 6500K 标准色温下表现良好，但在 4000K 暖色温显示器上会叠加出明显的黄色偏移，看起来像"旧纸张"而非"现代档案室"。

→ 建议提供一套冷色调备选色板，或将背景色微调为更中性的 `#F8F8F6`。

#### 视觉亮点

1. **Design Tokens 体系**：`style.qss` 头部的完整 token 文档（颜色、字体、间距）是我在此类项目中见过的最专业的设计文档之一。这应该被严格执行到每一行代码中。

2. **琥珀色单一强调色的克制**：整个系统只用 `#92400E` / `#B45309` 作为强调色，没有红绿蓝紫满天飞的乱象。这种克制让系统在视觉上显得成熟、有权威感。

3. **登录页的双栏布局**：左侧品牌区 + 右侧表单的经典布局执行得当，间距、字体层级、演示账号提示的设计都比较精致。

4. **Icon Rail 的 hover 颜色过渡动画**：150ms 的灰→琥珀色渐变是克制的微交互，既不干扰操作又提供了视觉反馈，是项目中最好的动效设计之一。
