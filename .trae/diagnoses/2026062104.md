## 🔍 身份一：资深代码审查专家报告

### 严重问题（必须修复）

**1. [PagesCommon.h] 巨型统一头文件 — 编译灾难**

`PagesCommon.h` 包含了 50+ 个 Qt 头文件和所有项目基础设施头文件，而几乎每个 `.cpp` 页面文件都包含它。这导致：
- 任何改动都触发近乎全量重编译
- 编译时间被人为膨胀 5-10 倍
- 真正的依赖关系被完全遮蔽

→ **修复方向**：每个 `.cpp` 文件仅包含自身实际使用的头文件。将 `PagesCommon.h` 拆分为按需引入的小头文件集合（已经部分存在于 `UiKit.h`、`AuthHelpers.h`、`DbHelpers.h`），或者直接删除它。

**2. [buildPropertyWorkorder.cpp:699] 单文件过长函数 + God Controller**

`WorkOrderPageController` 类承担了 UI 构建、SQL 查询构建、数据绑定、权限校验、对话框创建、通知发送等全部职责（699行）。`handleAction` 方法（第 322-555 行）包含 5 个 if-else 分支，每个分支内联创建完整对话框，代码高度重复。

→ **重构方向**：
- 将对话框创建提取为独立的 Dialog 类（`WorkOrderAssignDialog`、`WorkOrderFinishDialog`、`WorkOrderEvalDialog`）
- `handleAction` 使用策略模式或 switch+工厂分发
- `buildQuery()` 和 `executeQuery()` 可移入独立的 Repository/DAO 层

**3. [DashboardPage.cpp:37-38] SQL 拼接存在注入风险**

```cpp
const QString woScopeFilter = onlySelf ? QStringLiteral(" AND reporter_id = :scope_uid") : QString();
// 然后直接拼入 SQL:
"SELECT COUNT(*) FROM wo_work_order WHERE status IN (0,1,2,3) AND del_flag = 0%1"
    .arg(woScopeFilter)
```

虽然当前 `woScopeFilter` 只有两种固定值，但这种「字符串模板拼接 + 占位符」的模式极其脆弱。如果未来有人修改过滤条件为动态值，SQL 注入漏洞将悄然引入。`DashboardPage.cpp` 第 370-373 行的 `UNION ALL` 查询同理。

→ **修复方向**：统一使用 `DatabaseManager::query()` 的参数化接口，将 WHERE 条件构建逻辑封装到 `SqlCondition` 列表中，杜绝任何字符串拼接。

**4. [DashboardPage.cpp:62-64] QSqlQuery 未使用参数化绑定**

```cpp
QSqlQuery woPendQ;
woPendQ.prepare(QStringLiteral("SELECT COUNT(*) FROM wo_work_order WHERE ..."));
if (onlySelf) woPendQ.bindValue(":scope_uid", user.id);
```

这里虽然使用了 `prepare` + `bindValue`，但第 328 行的 `annQ` 查询：
```cpp
QSqlQuery annQ("SELECT COUNT(*) FROM nt_announcement WHERE del_flag = 0 AND date(publish_time) >= date('now', '-7 days')");
```
直接构造而非 prepare，虽当前无参数，但与项目其他地方的模式不一致，增加了未来修改时出错的风险。

→ **修复方向**：所有 SQL 查询统一走 `DatabaseManager::query(sql, params)` 接口。

**5. [AuthService.cpp:47] 硬编码 IP 地址**

```cpp
{"last_login_ip", "127.0.0.1"}
// ...
{"ip", "127.0.0.1"}
```

桌面应用确实运行在本地，但硬编码 `127.0.0.1` 使得登录日志毫无信息量。如果将来有任何远程访问场景（如通过 X11 转发或远程桌面），IP 信息将完全错误。

→ **修复方向**：使用 `QHostInfo::localHostName()` 或至少记录为 `"localhost"` 并标注来源。

**6. [buildSystemUser.cpp:336-344] N+1 查询问题**

在 `loadUsers` lambda 中，对每一行用户数据都发起一次角色查询：
```cpp
while (q.next()) {
    // ... 每个用户都执行:
    QSqlQuery rq(DatabaseManager::instance().database());
    rq.prepare("SELECT r.role_name FROM sys_role r ...");
    rq.bindValue(":uid", userId);
    rq.exec();
}
```

100 个用户 = 101 次 SQL 查询。

→ **修复方向**：使用 `JOIN` 一次性查询，或先批量查出所有用户角色映射到内存 `QMap<int, QStringList>`。

### 改进建议（强烈建议）

**7. [Utils.h] inline 函数定义在头文件中 — ODR 风险**

`Utils.h` 中 `hashPassword`、`verifyPassword`、`generateOrderNo` 等复杂函数全部定义为 `inline`。虽然 C++ 允许 inline 函数在多个翻译单元中出现，但这些函数包含复杂逻辑（PBKDF2、文件IO），不适合 inline。

→ **重构方向**：将实现移至 `Utils.cpp`，头文件仅保留声明。`NumericSortTableWidgetItem` 类同理。

**8. [Constants.h:645行] 枚举定义头文件过于庞大**

所有业务枚举（30+个命名空间）塞入单个头文件，每个枚举附带 `inline` 的 `labels()`/`colors()` 函数返回 `static const QMap`。这导致：
- 每次新增枚举都修改此文件，触发大面积重编译
- 头文件内大量静态局部变量，增加二进制体积

→ **重构方向**：按业务域拆分（`WorkOrderConstants.h`、`EventConstants.h` 等），或使用 X-Macro 减少样板代码。

**9. [PageFactory.cpp] if-else 链代替注册表**

103 行代码全是 `if (key == "xxx") return createXxxPage();`。新增页面必须手动添加，易遗漏。

→ **重构方向**：使用 `QMap<QString, std::function<BasePage*()>>` 注册表，配合静态初始化自动注册。

**10. [DatabaseManager.cpp:37-108] initSchema 依赖外部文件搜索**

`initSchema()` 在 5 个相对路径中搜索 `schema.sql`，部署时极易丢失文件。

→ **修复方向**：将 schema.sql 嵌入 Qt 资源文件（`.qrc`），从资源中读取，彻底消除路径依赖。

**11. [EventStatus 与 ComplaintStatus] 完全重复定义**

`Constants.h` 第 426-490 行，`EventStatus` 和 `ComplaintStatus` 的枚举值、标签、颜色完全相同。这是典型的 DRY 违反。

→ **修复方向**：`ComplaintStatus` 直接复用 `EventStatus`，或使用 `using ComplaintStatus = EventStatus;`。

### 重构方案

**A. 工单操作对话框统一提取**

当前问题：`buildPropertyWorkorder.cpp` 中 `handleAction` 方法内联创建 4 种不同对话框，代码高度重复（每个对话框都有相似的布局、按钮样式、信号连接模式）。

重构手法：提取方法 + 参数化对话框工厂

```cpp
// 重构后：独立的 WorkOrderDialogs.h
class WorkOrderDialogs {
public:
    static bool showAssignDialog(QWidget* parent, qint64 woId, int& outWorkerId);
    static bool showFinishDialog(QWidget* parent, qint64 woId, QString& outResult);
    static bool showEvalDialog(QWidget* parent, qint64 woId, int& outRating, QString& outContent);
    static bool showNewWorkOrderDialog(QWidget* parent, WorkOrderFormData& outData);
};
```

### 亮点

- `DatabaseManager` 的白名单表名校验 + 参数化绑定设计，安全意识到位
- `BasePage` 通过信号与 `MainWindow` 解耦的设计很好遵循了依赖倒置原则
- `UiKit` 命名空间提供了统一的 UI 组件工厂，减少重复
- 密码哈希使用 PBKDF2-SHA512 + 随机盐，远超同类项目的安全水准
- `PageCache` 的懒加载 + 缓存策略设计合理

### 偏离预警 🚨

**1. 技术驱动的 UI 复杂度偏离**

当前 UI 设计追求"现代档案室"美学风格（琥珀色 + 衬线标题 + 极细边框），视觉独特但可能偏离了目标用户群（社区工作者、中老年居民）的实际需求。这些用户更关心"字够大、按钮够明显、操作够简单"，而非设计感。

→ **影响**：核心用户学习成本增加
→ **纠偏建议**：在保持设计一致性的基础上，增加"大字体模式"或"简洁模式"切换选项

**2. 功能丰度 vs 核心体验的失衡**

项目覆盖了 10+ 个业务域、35+ 个页面，但部分核心流程的用户体验不够完善（如工单缺少房屋关联、缺少批量操作）。"广而不深"是当前最大风险。

→ **影响**：每个功能都"能用"但没有一个功能"好用"
→ **纠偏建议**：冻结新功能开发，集中资源打磨 TOP3 高频功能（工单、事件、通知）

**3. 单点部署风险**

SQLite 单文件数据库适合桌面演示，但真实部署场景中多终端并发访问、数据备份、灾难恢复等需求无法满足。

→ **影响**：项目无法从"课程实训"升级为"生产系统"
→ **纠偏建议**：抽象数据库访问层，预留 PostgreSQL/MySQL 切换能力

## 🎨 身份四：严苛视觉审查员报告

### 整体印象

- **一句话总评**：这是一个在 Qt 桌面框架内极力追求" editorial / archive "美学的项目，色彩克制、排版有意识、细节处理超出多数同类项目，但在某些页面上"设计感"反而成了可用性的敌人。
- **视觉评分：7.5 / 10**

### 严重视觉问题（让我想关掉页面）

**1. [登录页] 左侧品牌区过于暗沉**

`BrandPanel` 使用深蓝黑渐变背景 (`#0A1628` → `#0f172a` → `#001529`)，几何线条 alpha 仅 40，在常规 1080p 显示器上几乎不可见。建筑块的 alpha 15 更是近乎透明。整个左侧像一块死黑的面板，与右侧明亮的登录表单形成割裂感。

→ **原因分析**：设计师想营造"城市天际线"的高级感，但过度降低了视觉权重
→ **改进方向**：将几何线条 alpha 提升到 80-100，建筑块 alpha 提升到 30-50，让品牌区真正"看得见"

**2. [全局] 0px 圆角 + 1px 黑色边框 = 视觉攻击性过强**

整个应用几乎所有组件都是 0-2px 圆角，配合 `#141413` 的纯黑边框和深色面板，营造出一种冷硬、严肃的氛围。对于政务/管理系统这或许可以接受，但对于"社区"场景——一个本应温暖、亲和的产品——这种视觉语言传递的是"威严"而非"服务"。

→ **改进方向**：将卡片、对话框的圆角统一提升到 6-8px，按钮圆角提升到 4px。保留黑色面板的硬朗，但让数据区域更柔和。

**3. [工作台] 大数字指标区缺少视觉层次**

三个大数字指标（待处理工单、待审核事件、未读通知）并排显示，用 1px 垂直线分隔。问题是：
- 三个指标视觉权重完全相同，用户不知道该先看哪个
- 48px 衬线数字很有气势，但缺少颜色区分（全是 `#141413` 黑色），无法快速识别紧急程度
- "需要尽快响应" 的提示文字用 `#9A9A9A` 浅灰，几乎融入背景

→ **改进方向**：为"待处理工单"数字添加琥珀色或红色 tint 背景，使其成为视觉焦点。将提示文字颜色加深到 `#6B6B6B`。

### 视觉改进清单

**4. [表格] 表头过于压迫**

表头使用 `#141413` 纯黑背景 + `#FAF9F6` 白字 + 11px 字号 + letter-spacing: 1px + uppercase，视觉权威性过强。在"社区管理"场景中，这种军事化风格让人感到压抑。

→ **改进方向**：表头改为 `#F5F2EB` 浅暖灰背景 + `#141413` 深色文字，去掉 uppercase（中文场景下 uppercase 无意义）

**5. [按钮系统] 主按钮和普通按钮视觉差异不足**

`cssClass="primary"` 的主按钮（黑底白字）和普通按钮（白底黑边）在静态场景下差异不够明显，特别是当页面中同时存在多个按钮时。

→ **改进方向**：主按钮使用琥珀色（`#92400E`）背景，与整体暖色调协调，同时与普通按钮形成鲜明对比

**6. [工作台快捷操作] 图标缺少辨识度**

6 个快捷操作使用同色系（`#6B6B6B`）图标，hover 时通过 `QuickHoverFilter` 变为 `#B45309`。但在默认状态下，所有图标颜色相同，用户需要逐个阅读文字标签才能区分。

→ **改进方向**：为不同类别的快捷操作赋予微妙的色彩差异（如报修用琥珀、缴费用绿色、投诉用红色），提升扫视效率

**7. [对话框] 内联对话框缺少视觉区分**

工单操作（受理、派单、完成、评价）弹出的对话框使用相似的布局和样式，用户难以快速识别当前在执行哪种操作。

→ **改进方向**：为不同操作添加差异化的头部色带（受理=蓝色、派单=琥珀、完成=绿色、评价=紫色）

### 视觉亮点

- **色彩系统高度统一**：整个应用仅使用琥珀（`#92400E`/`#B45309`）作为唯一强调色，配合墨色（`#141413`）和暖象牙白（`#FAF9F6`），形成了清晰的视觉身份。这种克制在桌面应用中罕见。
- **QSS 样式表组织良好**：`style.qss` 按组件类型分区，有完整的设计令牌文档注释（第 1-39 行），维护性强。
- **动效系统克制得体**：200ms 淡入、600ms 数字滚动、150ms 图标颜色过渡 — 既提供了反馈，又不显花哨。
- **登录页表单设计**：右侧登录区域的排版、间距、输入框样式都很专业，"演示账号"提示条的设计简洁有效。
- **SVG 图标着色系统**：`tintSvgIcon` 函数实现了运行时图标着色，配合 hover 动画，为单调的图标增添了生命力。
