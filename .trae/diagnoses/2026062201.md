## 审查总览

经过三轮密集重构（C-A-B 闭环、第二轮巨型文件拆分、第三轮安全/架构/Controller 拆分）后，项目代码质量已有质的飞跃：`MainWindow` 从 1133 行拆分为 5 个辅助类，物业/治理/服务/统计/工作台等巨型页面已按子页面拆分，`DatabaseManager` 已形成白名单 + 参数化绑定的 SQL 注入防护体系，密码哈希采用 PBKDF2-SHA512 + 随机盐。

但重构也暴露了新的隐患，且仍有大量历史债务未清偿。本报告聚焦于 **当前版本真实存在的、未修复的、或重构过程中引入的新问题**。

## 🔍 身份一：资深代码审查专家报告

### 严重问题（必须修复）

#### 1. [buildPropertyWorkorder.cpp:694] WorkOrderPageController 内存泄漏

```cpp
void PageFactory::buildPropertyWorkorder(...) {
    auto *ctrl = new WorkOrderPageController(page, layout, table, db, emptyHint, page);
    ctrl->loadData();
}
```

`ctrl` 虽然将 `page` 设为 parent，但 `BasePage` 被 `PageCache` 缓存复用，生命周期由 `MainWindow` 管理；Controller 随页面创建而 `new`，但页面刷新时 `PageCache` 只是 `deleteLater` 旧页面。更关键的是：每次进入该页面都会 new 一个 Controller，而旧的 Controller 及其连接的 lambda/信号可能未正确释放。

→ **修复方向**：
- 将 Controller 的生命周期与 Page 绑定，在 Page 销毁时自动 delete Controller；或
- 使用 `std::unique_ptr` / `QPointer` 管理，并在 `BasePage` 析构时清理；或
- 将 Controller 提升为 `PageCache` 管理的成员，避免每次重建。

#### 2. [DashboardPage.cpp:63, 382] SQL 字符串模板拼接 — 注入风险模式

```cpp
const QString woScopeFilter = onlySelf ? QStringLiteral(" AND reporter_id = :scope_uid") : QString();
woPendQ.prepare(QStringLiteral("SELECT COUNT(*) FROM wo_work_order WHERE status IN (0,1,2,3) AND del_flag = 0%1")
                .arg(woScopeFilter));
```

虽然 `woScopeFilter` 当前只有两种固定值，但「模板字符串 + .arg() 拼接 WHERE 片段」的模式极易在未来维护中演变为 SQL 注入漏洞。第 382 行的 `UNION ALL` 动态查询同理。

→ **修复方向**：统一使用 `DatabaseManager::query()` 的参数化接口，或构建 `QList<SqlCondition>` 后由 `DatabaseManager::count()` / 自定义查询构建器生成 SQL。

#### 3. [buildSystemUser.cpp:337] 用户列表 N+1 查询

```cpp
while (q.next()) {
    qint64 userId = q.value(0).toLongLong();
    QSqlQuery rq(DatabaseManager::instance().database());
    rq.prepare("SELECT r.role_name FROM sys_role r ... WHERE ur.user_id = :uid");
    rq.bindValue(":uid", userId);
    rq.exec();
    while (rq.next()) { ... }
}
```

每渲染一行用户就执行一次角色查询。20 个用户 = 21 次 SQL，100 个用户 = 101 次。

→ **修复方向**：使用 `JOIN` 一次性查出用户+角色，或在加载前批量查询 `sys_user_role` 构建 `QMap<userId, QStringList>`。

#### 4. [PagesCommon.h] 巨型统一头文件仍未拆解

`PagesCommon.h` 仍包含 40+ 个 Qt 头文件和多个项目基础设施头文件。几乎所有 `.cpp` 页面文件都包含它，导致：
- 增量编译效率低下；
- 真实依赖关系被遮蔽；
- 违反 Include What You Use 原则。

→ **修复方向**：每个 `.cpp` 按需包含自身需要的头文件；将 `PagesCommon.h` 精简为 5-6 个真正通用的头文件，或逐步删除它。

#### 5. [Constants.h] 枚举文件臃肿 + EventStatus/ComplaintStatus 重复定义

`Constants.h` 已超过 650 行，集中了 30+ 个业务枚举。更严重的是 `EventStatus` 和 `ComplaintStatus` 的枚举值、标签、颜色完全相同（第 426-489 行），是典型 DRY 违反。

→ **修复方向**：
- 按业务域拆分为 `SysConstants.h`、`WoConstants.h`、`GeConstants.h` 等；
- `ComplaintStatus` 直接 `using ComplaintStatus = EventStatus;` 或复用同一套定义。

#### 6. [ArchivePage.cpp:616 / UiKit.cpp:561 / AIService.cpp:550 / EventPage.cpp:541 / WorkOrderDetailDialog.cpp:527] 拆分后仍存在巨型文件

虽然物业/治理/服务/统计页面已拆分，但以下文件仍超过 500 行，继续承担多个子功能：
- `ArchivePage.cpp`（616 行）：8 个基础档案子页面仍在同一文件；
- `UiKit.cpp`（561 行）：UI 组件工厂 + 表格/导出/搜索等杂项；
- `AIService.cpp`（550 行）：AI 对话 + 7 个上下文查询 + API Key 管理；
- `EventPage.cpp`（541 行）：事件列表 + 新建/编辑/详情/操作；
- `WorkOrderDetailDialog.cpp`（527 行）：详情展示 + 状态流转 + 评论/图片。

→ **修复方向**：继续按子功能或子 Tab 拆分，与已完成的 `buildPropertyXxx`、`buildGovernanceXxx`、`buildSystemXxx` 保持一致。

### 改进建议（强烈建议）

#### 7. [PageFactory.cpp:11-102] if-else 链注册表

103 行代码全是 `if (key == "xxx") return createXxxPage();`。新增页面必须手动添加，易遗漏，且与项目引以为傲的 `pageRegistry` 注册表理念不一致。

→ **修复方向**：使用 `QMap<QString, std::function<BasePage*()>>` 注册表，结合静态初始化或 `pageRegistry()` 自动生成 dispatcher。

#### 8. [DatabaseManager.cpp:39-43] initSchema 依赖外部文件路径搜索

```cpp
QApplication::applicationDirPath() + "/../sql/schema.sql",
QApplication::applicationDirPath() + "/sql/schema.sql",
"sql/schema.sql",
"../sql/schema.sql",
"../../sql/schema.sql"
```

部署时 schema.sql 极易因相对路径问题丢失，导致首次运行数据库为空。

→ **修复方向**：将 `schema.sql` 嵌入 Qt 资源文件（`.qrc`），从 `:/sql/schema.sql` 读取，彻底消除路径依赖。

#### 9. [AuthService.cpp:83, 100] 硬编码 IP 地址

```cpp
{"last_login_ip", "127.0.0.1"}
{"ip", "127.0.0.1"}
```

桌面应用运行在本地，但硬编码 IP 使登录日志无信息量，且与未来任何远程访问场景冲突。

→ **修复方向**：使用 `QHostInfo::localHostName()` 或至少记录 `"localhost"` 并标注来源。

#### 10. [Utils.h] 复杂函数仍 inline 在头文件中

`hashPassword`、`verifyPassword`、`generateOrderNo` 等包含复杂逻辑（PBKDF2、文件 IO）的函数仍定义为 `inline`，增加二进制体积和编译耦合。

→ **修复方向**：将实现移至 `Utils.cpp`，头文件仅保留声明。

#### 11. [buildPropertyWorkorder.cpp:322-555] handleAction 仍是 230 行 if-else 链

虽然 `loadData()` 已被拆分为三段式，但 `handleAction()` 仍内联创建 4 种不同对话框，每个分支包含完整 UI 构建 + 数据库操作 + 通知发送，代码高度重复。

→ **修复方向**：提取 `WorkOrderAssignDialog`、`WorkOrderFinishDialog`、`WorkOrderEvalDialog` 等独立对话框类；`handleAction` 使用策略模式或 switch+工厂分发。

#### 12. [DatabaseManager.cpp:68-108] initSchema 的分号解析器仍简陋

虽然已改进为忽略字符串字面量中的分号，但仍未处理：
- 多行注释 `/* ... */`；
- 行内注释 `--` 中的分号；
- 触发器/存储过程中的 `BEGIN...END` 块。

→ **修复方向**：使用更健壮的 SQL 拆分器，或至少跳过 `--` 注释行和 `/* */` 块。

#### 王主任 — 查看统计

1. **统计分析 → 综合看板** → 已有折线/柱状/饼图 → 体验良好。
2. **发现某事件超期** → 想督办 → 必须手动切换到「社区治理 → 督办管理」→ **痛点**：统计图表无法下钻或跳转到具体事件/工单详情。

### 偏离预警 🚨

**偏离 1：重构收益尚未完全覆盖到用户可见的体验层**
→ 代码架构已大幅改善，但居民报修后仍看不到工单号、列表仍默认展示全量、无法关联房屋。重构主要解决了"开发者维护难"，对"居民使用难"的改善有限。
→ **纠偏建议**：下一阶段将 30% 精力从代码拆分转向工单/事件核心流程的"最后一公里"打磨。

**偏离 2：桌面端定位与居民场景的根本矛盾未解决**
→ 智慧社区的核心用户（居民、维修工）主要使用移动端。当前纯桌面 SQLite 应用无法满足"随手报修"、"现场签到"需求。
→ **纠偏建议**：明确当前版本定位为"社区管理后台"，居民端功能仅作为演示数据入口，并标注"需移动端/H5 实现"。

### 战略建议

- **应聚焦于**：工单号反馈、居民默认"我的"视图、房屋关联、处理图片上传、统计图表下钻。
- **建议暂缓**：AI 智能问答增强、就业服务扩展、更多主题皮肤。
- **建议明确砍掉/标注**：居民端所有需要现场/移动场景的功能，改为管理端模拟展示。

## 🎨 身份四：视觉审查报告

### 整体印象

- **一句话总评**：「现代档案室」设计语言已建立较高一致性的基础框架，但操作列可点击性、卡片层次、圆角统一性等细节执行仍不到位，让专业感打了折扣。
- **视觉评分**：7.5 / 10

### 严重视觉问题

#### V-01 [全局] 表格操作列仍大量使用纯文本项

虽然工单页（`buildPropertyWorkorder.cpp`）已改为真实按钮，但投诉、访客、公告、督办、民意、停车、业委会议题等页面仍使用 `UiKit::createActionItem()` 返回的纯文本 `QTableWidgetItem`。这些文字与普通数据单元格视觉上没有本质区别，用户不会意识到可点击。

→ **修复方向**：所有操作列统一替换为 `UiKit::createActionCell()` + `QPushButton`，并设置 hover 下划线/背景变化。

### 视觉改进清单

| 页面/模块 | 具体问题 | 改进方向 |
|-----------|----------|----------|
| 全局 | `applyCardShadow()` 是空函数，卡片无阴影 | 使用嵌套 QFrame + 渐变背景或 QSS 模拟层次 |
| 全局 | border-radius 不统一（登录按钮 4px、工具栏 6px、表格 0px） | 严格按 design tokens 执行 |
| 登录页 | 品牌区建筑线条 alpha 仍偏低 | 提升至 80-120，建筑块 alpha 40-60 |
| Icon Rail | 仅图标无文字标签 | 扩展至 80px，图标下方加模块名缩写 |
| 对话框 | 标题颜色使用 `#0f172a` 而非全局 `#141413` | 统一颜色令牌 |

### 视觉亮点

1. **Design Tokens 文档化**：`style.qss` 头部的颜色/字体/间距规范仍是项目中最专业的设计资产。
2. **工单操作列按钮化**：第三轮重构后，工单列表操作按钮有 hover 反馈，可点击性显著提升。
3. **琥珀色强调色克制**：系统未出现多色混战，视觉权威感强。
4. **Icon Rail hover 渐变动画**：150ms 灰→琥珀过渡精致且不干扰。

## 📊 综合诊断总结

### Top 5 最高优先级改进项

| 优先级 | 编号 | 问题 | 影响面 | 改进建议 |
|--------|------|------|--------|----------|
| 🔴 P0 | 1 | WorkOrderPageController 内存泄漏 | 稳定性、资源占用 | 绑定 Controller 生命周期到 Page 或使用智能指针 |
| 🔴 P0 | 2 | DashboardPage SQL 模板拼接 | 安全性 | 统一使用参数化查询构建器 |
| 🔴 P0 | 3 | buildSystemUser N+1 查询 | 性能 | 批量 JOIN 或预加载角色映射 |
| 🟡 P1 | 4 | PagesCommon.h / Constants.h 臃肿 | 编译时间、可维护性 | 按需包含、按域拆分枚举 |
| 🟡 P1 | 5 | 全局操作列仍多为文本项 | 用户体验 | 统一替换为真实按钮 |

### 总体评价

项目已从"功能堆砌的实训项目"进化为"架构相对清晰的桌面应用"。当前最大的矛盾是：**代码侧的改进速度超过了产品体验的打磨速度**。下一步应将重心从"拆文件"转向"填功能坑"——尤其是工单/事件两条核心流程的闭环体验、居民视角的默认过滤、以及全局操作列的按钮化。只要把这些"最后一公里"补上，项目将从"能用"真正走向"好用"。
