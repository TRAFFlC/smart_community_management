#### 🔴 2. [src/pages/buildServiceEmployment.cpp:81-110] 分页计数与数据查询的 SQL 不一致

```cpp
QString sql = "SELECT id, title, company, ... FROM sv_job_posting WHERE del_flag = 0";
if (!searchText.isEmpty()) sql += " AND (title LIKE :search OR company LIKE :search)";
if (statusFilter >= 0) sql += " AND status = :status";
sql += " ORDER BY create_time DESC";
sql += " LIMIT :pageSize OFFSET :offset";

QVariantList cntBinds;
if (!searchText.isEmpty()) cntBinds << ":search" << ("%" + searchText + "%");
if (statusFilter >= 0) cntBinds << ":status" << (statusFilter);
cntBinds << ":pageSize" << pb->pageSize() << ":offset" << pb->offset();  // ❌
pb->setTotalCount(UiKit::executeCountQuery(sql, cntBinds));
```

**问题**：`UiKit::executeCountQuery` 内部用正则剥离 `LIMIT/OFFSET` 子句后包裹 `SELECT COUNT(*) FROM (...)`，但传入的 `cntBinds` 仍然包含 `:pageSize` 和 `:offset` 绑定参数。剥离后这两个占位符已不存在，SQLite 会忽略多余绑定（不报错），但这是**潜在的语义陷阱**——一旦 SQL 改写或正则失效，绑定数与占位符数不匹配就会抛异常。

**修复**：

```cpp
// 计数 SQL 不应包含分页参数
QString countSql = "SELECT id, title, company, salary_range, headcount, deadline, status "
                   "FROM sv_job_posting WHERE del_flag = 0";
if (!searchText.isEmpty()) countSql += " AND (title LIKE :search OR company LIKE :search)";
if (statusFilter >= 0) countSql += " AND status = :status";

QVariantList cntBinds;
if (!searchText.isEmpty()) cntBinds << ":search" << ("%" + searchText + "%");
if (statusFilter >= 0) cntBinds << ":status" << statusFilter;
pb->setTotalCount(UiKit::executeCountQuery(countSql, cntBinds));
```

#### 🔴 3. [src/pages/buildServiceEmployment.cpp:81-115] 同样的 Bug 模式遍布全项目

在 `buildPropertyInspection.cpp`、`buildPropertyComplaint.cpp`、`buildPropertyVisitor.cpp`、`buildPropertyBilling.cpp`、`buildServiceConvenience.cpp` 等至少 6 个文件中，均存在"把带 LIMIT/OFFSET 的 SQL 直接传给 executeCountQuery，并把分页参数一起塞进 cntBinds"的反模式。

**重构方向**：见后文"重构方案"。

#### 🔴 6. [src/widgets/MainWindow.cpp:340-345] switchPage 中 fadeInWidget 调用时机错误

```cpp
void MainWindow::switchPage(const QString &key) {
    m_currentPage = key;
    BasePage *basePage = m_pageCache->getOrCreatePage(key);
    QWidget *page = basePage ? static_cast<QWidget *>(basePage) : m_pageCache->pageWidget(key);
    if (page) {
        m_contentStack->setCurrentWidget(page);
        UiKit::fadeInWidget(page, 200);  // ❌ 每次 switchPage 都触发淡入
    }
    // ...
}
```

**问题**：每次切换页面（包括从同一页面返回）都触发 200ms 淡入动画，且 `fadeInWidget` 内部使用 `QGraphicsOpacityEffect`。在 Qt6 上，对已设置 GraphicsEffect 的 widget 反复 setGraphicsEffect 会导致**内存增长**和**偶发闪烁**。`applyCardShadow` 注释中已明确提到 Qt6 的 painter 冲突问题，但这里仍踩坑。

**修复**：仅在首次创建页面时淡入，或用 QPropertyAnimation + windowOpacity 替代。

#### 🔴 7. [src/widgets/PageCache.cpp:46-52] refreshPage 中的 deleteLater + 立即重建存在悬挂指针风险

```cpp
BasePage *PageCache::refreshPage(const QString &key) {
    auto it = m_pages.find(key);
    if (it != m_pages.end()) {
        QWidget *oldPage = it.value();
        if (oldPage && m_stack) m_stack->removeWidget(oldPage);
        if (oldPage) oldPage->deleteLater();  // ❌ 异步删除
        m_pages.erase(it);
    }
    return getOrCreatePage(key);  // 立即创建新页面
}
```

**问题**：`deleteLater` 是异步的，事件循环下一次迭代才真正删除。但 `getOrCreatePage` 会立即 `new` 一个新页面并 `addWidget`。如果旧页面的析构信号（如 destroyed）连接了其他控件，新页面可能收到旧页面的信号。此外，`m_stack->removeWidget` 后旧 widget 仍可能短暂可见。

**修复**：用 `delete oldPage`（同步）替代 `deleteLater`，或在 deleteLater 后用 `QTimer::singleShot(0, ...)` 延迟创建。

### 重构方案

#### 重构 1：统一分页查询模式

**问题**：全项目 10+ 处分页查询，每处都手写 SQL + LIMIT/OFFSET + 计数 SQL，Bug 频出。

**重构手法**：抽取 `PagedQuery` 工具类

```cpp
// src/ui_kit/PagedQuery.h
class PagedQuery {
public:
    PagedQuery(const QString& baseSql, const QVariantMap& filters = {});

    // 设置 LIKE 过滤（自动加 AND）
    PagedQuery& like(const QString& column, const QString& value);

    // 设置等值过滤
    PagedQuery& eq(const QString& column, const QVariant& value);

    // 执行查询，返回 (rows, totalCount)
    struct Result {
        QList<QSqlRecord> rows;
        int totalCount = 0;
    };
    Result execute(int pageSize, int offset);

private:
    QString m_baseSql;        // 不含 LIMIT/OFFSET
    QVariantMap m_filters;
    QStringList m_whereParts;
};
```

**调用方对比**：

```cpp
// 重构前（buildServiceEmployment.cpp，30 行）
QString sql = "SELECT ... FROM sv_job_posting WHERE del_flag = 0";
if (!searchText.isEmpty()) sql += " AND (title LIKE :search OR company LIKE :search)";
if (statusFilter >= 0) sql += " AND status = :status";
sql += " ORDER BY create_time DESC LIMIT :pageSize OFFSET :offset";
QVariantList cntBinds;
// ... 10 行绑定逻辑 ...
pb->setTotalCount(UiKit::executeCountQuery(sql, cntBinds));
jobQ.prepare(sql);
// ... 10 行绑定逻辑 ...

// 重构后（5 行）
PagedQuery q("SELECT ... FROM sv_job_posting WHERE del_flag = 0");
if (!searchText.isEmpty()) q.like("title", searchText).like("company", searchText);
if (statusFilter >= 0) q.eq("status", statusFilter);
auto result = q.execute(pb->pageSize(), pb->offset());
pb->setTotalCount(result.totalCount);
for (const auto& row : result.rows) { /* 填表 */ }
```

### 偏离预警 🚨

#### 🚨 偏离 1：AI 智能问答投入过重，核心场景价值低

**现状**：`AIService` + `buildSystemAI` 投入大量精力实现 OpenRouter 接入、密钥加密、上下文构建、多模型切换。

**问题**：
- AI 问答是 P2 需求，但代码量占比远超 P0 需求（如工单图片上传）
- 依赖外网，社区网络环境可能不稳定
- 居民真正需要的是"一键报修"，不是"问 AI 怎么报修"
- API Key 管理复杂，课程实训场景下维护成本高

**纠偏建议**：将 AI 降级为"知识库检索"，移除 LLM 调用，保留 `ai_knowledge` 表的 FAQ 查询。

#### 🚨 偏离 2：便民服务仅列表展示，未形成闭环

**现状**：`buildServiceConvenience` 只展示服务订单列表，无下单入口。

**问题**：
- 居民无法在系统中下单便民服务（如开锁、保洁）
- 服务商无法接单
- 订单状态无法流转
- 这是"半成品"功能，不如不展示

**纠偏建议**：要么补全下单 → 接单 → 服务 → 评价闭环，要么从菜单移除，避免用户困惑。

#### 🚨 偏离 3：考核管理无自动计算，形同虚设

**现状**：`buildGovernanceAssessment` 可配置考核指标和查看结果，但结果需手动录入。

**问题**：
- 考核的本质是"自动从工单/事件中统计数据"
- 当前设计把考核做成了"手动打分表"，价值极低
- 管理者不会用一个手动录入的考核系统

**纠偏建议**：实现"考核指标 → SQL 自动统计 → 结果生成"的自动化链路。

### 模拟需求变更

#### 变更 1：居民端微信小程序报修

**需求描述**：居民无需下载 App，微信扫码即可报修，进度通过微信通知。

### 严重视觉问题（让我想关掉页面）

#### ❌ 1. [LoginWidget.cpp:30-60] 登录页左侧品牌区配色与全局设计语言冲突

**问题**：登录页用深蓝渐变（`#0A1628` → `#0f172a` → `#001529`）+ 琥珀色线条，但全局 QSS 定义的设计语言是"暖象牙白 + 墨色 + 琥珀"。登录页像是**另一个项目**的页面。

**原因**：登录页独立绘制，未遵循 `style.qss` 的 Design Tokens。

**改进**：登录页应使用 `#FAF9F6` 底 + `#141413` 文字 + `#92400E` 强调，与主应用一致。

---

#### ❌ 2. [buildSystemAI.cpp:40-50] AI 助手页面用 `#f5f5f5` 背景，与全局 `#FAF9F6` 不一致

```cpp
chatDisplay->setStyleSheet(R"(
    QTextEdit {
        background: #f5f5f5; border: 1px solid #e2e8f0; ...
    }
)");
```

**问题**：`#f5f5f5` 是冷灰色，与全局暖象牙白 `#FAF9F6` 色温不同，并排时会有"脏"的感觉。`#e2e8f0` 也是冷灰边框，与全局 `#D4D0C8` 暖灰冲突。

**改进**：统一使用 `#FAF9F6` 和 `#D4D0C8`。

---

#### ❌ 3. [buildServiceVolunteer.cpp:55-70] 统计卡片用 `border-radius:6px`，全局规范是 `2px`

```cpp
card->setStyleSheet("QFrame{background:#fff;border-radius:6px;...}");
```

**问题**：`style.qss` 明确规定"圆角：全局 2px（克制，非 4-8px）"，但 10+ 个页面中的卡片都用 `6px`。这是**设计规范未落地**的典型表现。

**改进**：全局替换为 `border-radius:2px`，或在 UiKit 中集中定义。

---

#### ❌ 4. [buildSystemAI.cpp:75-80] 快捷问题按钮用 `#f0f5ff` / `#adc6ff`，不在调色板内

```cpp
btn->setStyleSheet("QPushButton{background:#f0f5ff;color:#b45309;border:1px solid #adc6ff;...}");
```

**问题**：`#f0f5ff` 和 `#adc6ff` 是 Ant Design 的蓝色系，与"现代档案室"的琥珀+墨色体系**完全无关**。这种"随手抄来的颜色"是视觉一致性的最大敌人。

**改进**：使用 `rgba(146, 64, 14, 0.06)` 底 + `#92400E` 文字 + `#D4D0C8` 边框。

#### ❌ 5. [buildReportDashboard.cpp:55-60] 统计卡用 6 种颜色，违反"单一琥珀强调"原则

statsRow->addWidget(mkStatCard("工单总数", ..., "#b45309", ...));

statsRow->addWidget(mkStatCard("事件总数", ..., "#15803d", ...));

**改进**：全部用 `#141413` 文字 + `#92400E` 数值，仅用图标区分类型。

## 📊 综合诊断总结

### Top 5 最高优先级改进

| 优先级 | 问题 | 影响 | 工作量 |
|--------|------|------|--------|
| **P0** | 修复 TodoCenterPage 事件计数 Bug（`woQ` → `evQ`） | 管理者看到错误数据，决策失误 | 1 行 |
| **P0** | 修复分页计数 SQL 与绑定参数不匹配的系统性问题 | 潜在 SQL 异常，搜索/筛选可能失效 | 中（需重构 PagedQuery） |
| **P0** | 修复居民数据权限：居民登录后应只看自己的工单/事件 | 隐私泄露，居民看到全部数据 | 中 |
| **P1** | 统一视觉规范：消除 6+ 处 `border-radius:6px` 和冷灰色滥用 | 视觉不一致，降低专业感 | 小（全局替换） |
| **P1** | 补全工单图片上传功能 | 核心场景缺失，居民报修无现场照片 | 中 |

### 架构层面的根本性建议

1. **引入 Controller 层**：参照 `WorkOrderPageController`，为每个子页面抽取 Controller，消除内联 lambda 捕获裸指针的脆弱模式。

2. **统一分页查询**：抽取 `PagedQuery` 工具类，消除 10+ 处手写 SQL + LIMIT/OFFSET 的重复与 Bug。

3. **设计规范落地**：在 `UiKit` 中集中定义 `ColorPalette`、`Spacing`、`Radius` 常量，禁止在页面代码中硬编码颜色字符串。

4. **页面注册单一数据源**：合并 `MainWindow::pageRegistry` 和 `PageFactory::createPage` 的两处 registry，从 `sys_menu` 表动态加载。

5. **移动端策略**：明确"桌面端不适配移动端"的产品边界，移动需求通过微信小程序或 Web 端承接，避免在 Qt Widgets 中强行响应式。

### 结语

这个项目在**架构骨架**（DatabaseManager 安全查询、AuthService 权限体系、PageCache 缓存、BasePage 信号解耦）上展现了不错的工程素养，在设计语言上有明确的审美追求。但在**执行细节**上（复制粘贴 Bug、分页 SQL 系统性问题、视觉规范未落地、功能未闭环）暴露了"重架构轻实现"的通病。

建议按 P0 → P1 的顺序逐项修复，**先修 Bug，再补功能，最后优化视觉**。不要在 P0 Bug 未修的情况下继续新增功能——那只会让技术债雪球越滚越大。

> "代码不会自己变好，但会自己变坏。" — 审视者
