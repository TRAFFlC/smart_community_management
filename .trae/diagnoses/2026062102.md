## 🔍 身份一：代码审查报告

### 严重问题（必须修复）

#### 1. [GovernancePage.cpp] 巨型函数 — 单文件 1168 行，createGovernancePage 函数体约 1160 行

这是典型的「上帝函数」（God Function）。一个函数承担了事件、巡查、关怀、督办、民意、考核 6 个子页面的全部构建逻辑，可读性为零，维护成本极高。

→ **修复建议**：按 `sub` 类型拆分为独立函数或文件，与 Property 子页面保持一致的架构。

```cpp
// 重构方向：每个子页面一个独立函数
BasePage* createGovernanceEventPage();
BasePage* createGovernanceInspectionPage();
BasePage* createGovernanceCarePage();
BasePage* createGovernanceSupervisionPage();
BasePage* createGovernanceOpinionPage();
BasePage* createGovernanceAssessmentPage();

// createGovernancePage 退化为路由分发
BasePage* PageFactory::createGovernancePage(const QString& sub) {
    if (sub == "event")       return createGovernanceEventPage();
    if (sub == "inspection")  return createGovernanceInspectionPage();
    if (sub == "care")        return createGovernanceCarePage();
    if (sub == "supervision") return createGovernanceSupervisionPage();
    if (sub == "opinion")     return createGovernanceOpinionPage();
    if (sub == "assessment")  return createGovernanceAssessmentPage();
    return createPlaceholderPage();
}
```

#### 2. [buildPropertyWorkorder.cpp:55-185] 闭包嵌套过深、lambda 捕获混乱

`loadWorkOrders` lambda 捕获了 `table, searchEdit, filterCombo, myTasksCheck, actionHandlerPtr, emptyHint, pb` 共 7 个变量，且内部逻辑包含 SQL 拼接、参数绑定、数据渲染、权限判断四层嵌套。`handleWorkOrderAction` lambda 又反过来捕获 `loadWorkOrders`，形成循环依赖，需要 `shared_ptr<function>` 这种丑陋手段绕过。

→ **修复建议**：将页面逻辑封装为独立的 `WorkOrderPageController` 类，用成员变量代替 lambda 捕获，用信号/槽代替循环引用。

```cpp
class WorkOrderPageController : public QObject {
    Q_OBJECT
public:
    WorkOrderPageController(BasePage* page, QTableWidget* table, ...);
    void loadData();
    void handleAction(qint64 id, int status);
private:
    BasePage* m_page;
    QTableWidget* m_table;
    // ...
};
```

#### 3. [GovernancePage.cpp:295-332] 关怀对象列表缺少分页

`loadGroups` 函数没有 `LIMIT/OFFSET` 分页，数据量大时会一次性加载所有记录。其他子页面（巡查、走访、督办等）都有分页支持，唯独这里遗漏。

→ **修复建议**：添加 `PaginationBar`，与其他页面保持一致。

#### 4. [buildPropertyWorkorder.cpp:239-250] SQL 查询未使用参数化绑定

维修人员查询使用了硬编码 SQL 字符串：
```cpp
QSqlQuery workerQ("SELECT u.id, u.nickname, u.username FROM sys_user u JOIN sys_user_role ur ON ...");
```
没有调用 `prepare()` + `bindValue()`。虽然当前查询没有外部输入，但这种写法一旦引入动态条件就容易产生 SQL 注入风险，且与项目整体「参数化绑定」的安全规范不一致。

→ **修复建议**：统一使用 `prepare()` + `bindValue()`。

#### 5. [MainWindow.cpp:104-105] MainWindow 裸指针 + 手动内存管理

```cpp
mainWindow = new MainWindow();
// ...
delete mainWindow;
```

`mainWindow` 是裸指针，虽然有 `delete`，但如果 `app.exec()` 异常退出或事件循环中发生未捕获异常，可能导致内存泄漏。

→ **修复建议**：使用 `std::unique_ptr<MainWindow>` 或让 Qt 对象树管理生命周期。

#### 6. [DatabaseManager.cpp:68] Schema 初始化 SQL 解析过于简陋

```cpp
QStringList statements = sql.split(';', Qt::SkipEmptyParts);
```

简单的 `;` 分割无法处理 SQL 中包含分号的字符串字面量（如 `INSERT INTO ... VALUES ('a;b')`）。虽然当前 schema.sql 没有这种情况，但未来可能踩坑。

→ **修复建议**：使用更健壮的 SQL 语句分割器，或至少跳过注释行中的分号。

#### 7. [GovernancePage.cpp 多处] 状态判断使用硬编码数字

```cpp
// line 132
QString planStsText = pSts == 0 ? ... : pSts == 1 ? ... : ...;
// line 230
auto *stsItem = UiKit::createTagTableItem(iSts == 0 ? ... : ...);
```

这些硬编码数字应该使用 `Constants.h` 中定义的枚举值。

→ **修复建议**：使用已定义的枚举常量，如 `InspectionPlanStatus::Pending` 等。如果对应枚举不存在，应先在 `Constants.h` 中补充。

### 改进建议（强烈建议）

#### 1. [PagesCommon.h] 巨型统一头文件 — 包含 50+ 个 Qt 头文件

每个页面源文件都包含 `PagesCommon.h`，而它引入了几乎所有 Qt 头文件。这会导致编译时间显著增加，且隐藏了各页面实际依赖关系。

→ **重构方向**：移除 `PagesCommon.h`，各页面按需包含所需头文件。或使用 C++20 modules。

#### 2. [MainWindow.cpp:601-655] `moduleSections` 静态映射与 `pageRegistry` 重复

`buildSectionPanel` 中的 `moduleSections` 映射与 `pageRegistry` 存在大量重复数据。页面注册信息应该只有一个真实来源（Single Source of Truth）。

→ **重构方向**：从 `pageRegistry` 自动生成 section 列表，消除手动同步。

#### 3. [MainWindow.cpp:783-796] `switchPage` 中的模块归属判断使用字符串前缀

```cpp
if (key.startsWith("2"))
    newModule = QStringLiteral("基础档案");
```

这种硬编码前缀判断非常脆弱，key 命名规则一旦变化就会崩溃。

→ **重构方向**：在 `PageInfo` 结构体中添加 `moduleKey` 字段，从注册表直接查询模块归属。

#### 4. [Utils.h:14-21] 匿名命名空间中的 `nextSequenceNumber` 定义在头文件中

```cpp
namespace {
    inline int nextSequenceNumber() {
        static QAtomicInt counter(0);
        return counter.fetchAndAddRelaxed(1) + 1;
    }
}
```

匿名命名空间在头文件中意味着每个包含该头文件的翻译单元都有自己的 `counter` 实例，不能保证全局唯一性。虽然当前只有一个 `main.cpp` 包含链使用它，但这是潜在隐患。

→ **修复建议**：移到 `Utils.cpp` 中实现，头文件只保留声明。

#### 5. [buildPropertyWorkorder.cpp:411] `cellEntered` 信号未设置 `mouseTracking`

```cpp
QObject::connect(table, &QTableWidget::cellEntered, page, [=](int r, int c)
        { table->viewport()->setCursor(c == 7 ? Qt::PointingHandCursor : Qt::ArrowCursor); });
```

`cellEntered` 信号需要 `setMouseTracking(true)` 才能触发。当前代码未显式设置，依赖 Qt 默认行为可能在某些版本中不生效。

→ **修复建议**：在表格初始化时添加 `table->setMouseTracking(true);`。

#### 6. [GovernancePage.cpp:746-753] 民意收集统计卡片查询缺少参数化

4 条独立的 `QSqlQuery` 直接在构造时执行，每次切换页面都会重新查询。

→ **修复建议**：合并为单条 SQL，并在页面切换时使用缓存。

### 重构方案

#### 问题概述：GovernancePage 巨型函数 → 拆分文件

当前 `GovernancePage.cpp` 1168 行，是所有页面中最长的。与 Property 子页面已经拆分为 `buildPropertyWorkorder.cpp` 等独立文件的做法不一致。

→ **重构手法**：Extract Function → Extract File

```
GovernancePage.cpp (1168 行)
├── buildGovernanceEvent.cpp        (已有 EventPage.cpp，可复用)
├── buildGovernanceInspection.cpp   (提取巡查计划+记录 Tab)
├── buildGovernanceCare.cpp         (提取关怀对象+走访记录 Tab)
├── buildGovernanceSupervision.cpp  (提取督办管理)
├── buildGovernanceOpinion.cpp      (提取民意收集)
└── buildGovernanceAssessment.cpp   (提取考核管理)
```

#### 问题概述：lambda 循环依赖 → Controller 模式

当前多个页面使用 `shared_ptr<function>` 解决 lambda 循环引用：

```cpp
auto actionHandlerPtr = std::make_shared<std::function<void(qint64, int)>>();
std::function<void()> loadWorkOrders = [...] { ... };
std::function<void(qint64, int)> handleAction = [loadWorkOrders] { ... };
*actionHandlerPtr = handleAction;
```

→ **重构手法**：Replace Lambda with Controller Class

```cpp
// Before:
auto ptr = std::make_shared<std::function<void(qint64, int)>>();

// After:
class PageController {
public:
    void loadData();
    void handleAction(qint64 id, int status);
    // 内部持有 table/searchEdit 等引用
};
```

### 亮点（值得肯定）

1. **DatabaseManager 的安全设计**：白名单表名校验、标识符校验、操作符白名单、参数化绑定，形成了完整的 SQL 注入防护体系。
2. **密码安全**：PBKDF2-SHA512 + 16 字节随机盐 + 10000 轮迭代，并保留旧版兼容，安全意识到位。
3. **统一页面注册表**（`pageRegistry`）：面包屑、导航、页面创建从单一数据源派生，消除了多处维护的隐患。
4. **BasePage 信号解耦**：页面通过 `sendNotificationRequested` / `navigateToRequested` 信号与 MainWindow 通信，不直接依赖宿主窗口。
5. **Constants.h 枚举体系**：`EnumHelper` 模板 + 命名空间模式，类型安全且向后兼容。
6. **UiKit 动效系统**：fadeIn、slideIn、countUp、pulse 等动画效果克制且有品位，150-300ms 时长恰到好处。

### 偏离预警 🚨

1. **Qt6::Charts 引入但未使用** → CMakeLists.txt 链接了 `Qt6::Charts`，但统计页面全部是纯数字卡片，没有任何图表。这说明开发资源投入在页面数量扩展上，而忽略了核心功能的深度打磨。→ **影响**：用户感知系统"能用但不好用"。→ **纠偏建议**：优先在综合看板和工单/事件统计页面添加图表。

2. **Demo 数据过于丰富** → `DemoDataService` 有 25+ 个初始化函数，几乎为每张表都填充了演示数据。这占用了大量开发精力，但演示数据的真实性难以保证。→ **影响**：可能出现数据关联不一致。→ **纠偏建议**：精简 Demo 数据，只覆盖核心流程路径。

3. **页面拆分不均衡** → Property 子页面已拆分为 9 个独立文件，但 Governance 仍是 1168 行的单文件。→ **影响**：维护成本高，新功能开发时容易冲突。→ **纠偏建议**：统一拆分策略。

### 模拟需求变更

#### 新需求 1：居民端微信小程序

**合理性**：高。居民不可能每天打开桌面端报修，移动端是刚需。

**现有架构承接能力**：当前系统是纯桌面端 SQLite 本地数据库，没有后端 API 层。如果要支持小程序，需要：
- 新增 RESTful API 服务层
- 数据库迁移到服务端（PostgreSQL/MySQL）
- 前端桌面端改为连接远程 API

**评估**：当前架构**无法直接承接**，需要在架构上做重大调整。建议从现在开始抽象 Service 层接口，为未来 C/S 分离做准备。

#### 新需求 2：AI 智能分单

**合理性**：中高。根据工单类型/位置自动分配给对应维修人员。

**现有架构承接能力**：当前 `AIService` 已有知识库和对话功能基础。工单分单可以在 `handleWorkOrderAction` 的"派单"逻辑中添加推荐算法。现有架构可以承接。

#### 新需求 3：多小区数据隔离

**合理性**：高。物业公司管理多个小区时，数据需要严格隔离。

**现有架构承接能力**：`buildDataScopeFilter` 已实现了基于 `estate_id` 的数据范围过滤，架构上已预留了多小区隔离的基础。但需要确保所有查询都经过此过滤器，当前 GovernancePage 中部分查询**未使用** `buildDataScopeFilter`。

### 战略建议

- **应聚焦于**：统计分析图表化、工单/事件 SLA 超时预警、实时消息推送。这些是让用户"用起来"的关键。
- **建议暂缓**：智能问答增强、就业服务扩展。这些是锦上添花但非雪中送炭。
- **建议砍掉/重构**：`PagesCommon.h` 巨型头文件，编译效率问题会随项目增长恶化。

## 🎨 身份四：视觉审查报告

### 整体印象

- **一句话总评**：这是一套有着明确设计语言（现代档案室/编辑期刊风）的系统，色彩克制、排版有序，但在细节执行上暴露出"设计系统建了一半"的问题 — 全局 QSS 和局部内联样式之间存在明显割裂。
- **视觉评分**：7 / 10

### 严重视觉问题（让我想关掉页面）

#### 1. [全局] 圆角不一致 — 设计系统自相矛盾

`style.qss` 注释明确写着「圆角: 全局 2px（克制，非 4-8px）」，但实际执行中：

- QSS 全局按钮：`border-radius: 2px` ✅
- 登录页表单面板：`border-radius: 6px` ❌（内联样式覆盖）
- 工具栏容器：`border-radius: 6px` ❌（多处内联）
- 统计卡片：`border-radius: 6px` ❌（GovernancePage 内联）
- 提示框：`border-radius: 6px` ❌（LoginWidget hintFrame）

→ **原因分析**：QSS 全局规则被各页面构建函数中的 `setStyleSheet()` 内联样式覆盖。开发者在不同时间点写了不同圆角值。
→ **改进方向**：在 QSS 中定义 `.card`、`.toolbar` 等语义化样式类，页面通过 `setProperty("cssClass", "card")` 引用，禁止内联覆盖。

#### 2. [GovernancePage] 统计卡片与 QSS 主题严重脱节

GovernancePage 中的统计卡片使用了蓝色系配色（`#e6f4ff`、`#2563eb`），而项目设计系统的颜色只有琥珀色 `#92400E` 和墨色 `#141413`。

```cpp
// GovernancePage.cpp line 134-137 — 蓝色系与琥珀主题冲突
QColor planStsBg = pSts == 0 ? QColor("#eff6ff") : pSts == 1 ? QColor("#fffbeb") : QColor("#f0fdf4");
QColor planStsFg = pSts == 0 ? QColor("#2563eb") : pSts == 1 ? QColor("#d97706") : QColor("#15803d");
```

→ **改进方向**：状态颜色应统一使用 Constants.h 中定义的颜色映射，而非各页面自行硬编码。

#### 3. [LoginWidget] 登录页品牌区「用力过猛」

品牌区的几何线条建筑感设计本身不错，但存在以下问题：
- 品牌文字使用 `Microsoft YaHei UI` 硬编码，而非全局配置的 Noto 字体，与设计系统脱节
- 建筑块的 alpha 值（15-40）在高分屏上几乎看不见
- 底部「课程实训项目」字样暴露了项目的教学性质，降低了专业感

### 视觉改进清单

| 页面/模块 | 具体问题 | 改进方向 |
|-----------|----------|----------|
| 全局 | 工具栏背景色 `#ffffff` + 边框 `#e2e8f0` 与 QSS 主题色 `#FAF9F6` + `#D4D0C8` 不一致 | 统一使用 QSS 变量或主题常量 |
| 全局 | `toolbar` 容器使用了 `border-radius: 6px` 内联样式，与 QSS 2px 圆角冲突 | 改用 QSS 类选择器 |
| GovernancePage | 状态标签颜色使用蓝色系，与琥珀主题冲突 | 使用 Constants.h 中的颜色映射 |
| 统计卡片 | 多处 `createMiniCard`/`createStatCard` lambda 重复定义 | 提取为 UiKit 公共组件 |
| 对话框 | 新建/编辑对话框标题使用 `#0f172a`（Tailwind slate-900），与全局 `#141413` 不一致 | 统一颜色令牌 |
| 表格 | 部分表格设置了 `setStyleSheet(UiKit::TABLE_STYLE)` 内联覆盖全局 QSS | 移除内联，依赖全局 QSS |
| 滚动条 | QSS 定义了 8px 宽度，但 Icon Rail 区域用 `width: 0` 隐藏了滚动条 | 考虑是否统一 |

### 视觉亮点

1. **「现代档案室」设计语言执行到位**：暖象牙白底 `#FAF9F6` + 近黑墨色 `#141413` + 琥珀强调 `#92400E` 的三色体系贯穿全局，视觉权威感强。
2. **登录页品牌区**：几何线条+建筑轮廓的自定义绘制（`BrandPanel::paintEvent`）体现了较高的设计追求，虽然 alpha 值偏低但概念出色。
3. **Icon Rail 导航**：64px 深色侧边栏 + hover 琥珀色渐变动画，交互感精致。
4. **表格表头设计**：`#141413` 深色表头 + 11px 大写字母 + 1px letter-spacing，专业感强。
5. **Toast 提示**：左侧琥珀色竖条 + 墨色背景 + 滑入动画，细节考究。

## 📊 综合诊断总结

### Top 5 最高优先级改进项

| 优先级 | 改进项 | 影响维度 | 预计工作量 |
|--------|--------|----------|------------|
| 🔴 P0 | **GovernancePage.cpp 拆分**（1168 行巨型函数 → 6 个独立文件） | 代码质量 + 可维护性 | 2-3 天 |
| 🔴 P0 | **统计分析页面添加图表**（使用已引入的 Qt6::Charts） | 用户体验 + 需求完成度 | 3-5 天 |
| 🟡 P1 | **统一圆角/颜色系统**（消除内联样式与 QSS 主题的冲突） | 视觉一致性 | 1-2 天 |
| 🟡 P1 | **Lambda 循环依赖重构为 Controller 模式** | 代码质量 + 可测试性 | 3-5 天 |
| 🟢 P2 | **工作台角色个性化**（不同角色看到不同的首页布局和数据） | 用户体验 | 3-5 天 |

### 总体评价

这是一个**完成度相当高**的课程实训项目。数据库设计（30+ 张表、完善的索引和约束）、安全机制（PBKDF2 密码哈希、SQL 注入防护、DPAPI 加密）、设计系统（现代档案室风格 QSS）三个维度都展现了超出课程水平的追求。

但项目目前最大的问题是**「做了很多但做不均匀」** — 页面数量覆盖了 7 大模块 30+ 子页面，但每个页面的深度打磨不足；Qt6::Charts 引入了却没用；视觉设计系统定义了但执行不一致。建议下一阶段**从广度扩展转向深度打磨**，优先解决统计分析可视化、GovernancePage 拆分、和视觉一致性三个问题。
