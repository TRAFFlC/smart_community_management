### 🔍 代码审查报告

#### 严重问题（必须修复）

**1. 巨型单文件 — God Function 泛滥**

[PropertyPage.cpp](file:///d:/smart_community_management/src/pages/PropertyPage.cpp) 只有56行是壳，真正逻辑全部在 [buildPropertySubPages.cpp](file:///d:/smart_community_management/src/pages/buildPropertySubPages.cpp)；[GovernancePage.cpp](file:///d:/smart_community_management/src/pages/GovernancePage.cpp) **1738行**、[SystemPage.cpp](file:///d:/smart_community_management/src/pages/SystemPage.cpp) **1866行**、[DemoDataService.cpp](file:///d:/smart_community_management/src/services/DemoDataService.cpp) **1618行**、[DashboardTodoMessagePage.cpp](file:///d:/smart_community_management/src/pages/DashboardTodoMessagePage.cpp) **991行**。这些文件严重违反单一职责原则，每个文件实质上是一个"模块宇宙"。

→ **修复方向**：将每个 `createXxxPage` 工厂函数拆为独立子类（如 `WorkOrderPage : BasePage`），每个子类拥有自己的 `loadData()`、`buildToolbar()`、`buildTable()` 等方法。目标：**每个文件不超过 400 行**。

**2. UI 层直接嵌入 SQL — 架构层次混乱**

所有页面中 SQL 查询直接写在了 UI 构建的 lambda 里。例如 [GovernancePage.cpp:93](file:///d:/smart_community_management/src/pages/GovernancePage.cpp#L93):

```cpp
QString sql = "SELECT id, event_no, title, event_category, priority, status, reporter_name, create_time, reporter_id, assign_to FROM ge_event WHERE del_flag = 0";
```

[DashboardTodoMessagePage.cpp:65](file:///d:/smart_community_management/src/pages/DashboardTodoMessagePage.cpp#L65):

```cpp
woPendQ.prepare(QStringLiteral("SELECT COUNT(*) FROM wo_work_order WHERE status IN (0,1,2,3) AND del_flag = 0%1").arg(woScopeFilter));
```

整个项目约有 **100+ 处** SQL 直接散落在 UI 代码中，完全没有 Service/Repository 分层。

→ **修复方向**：为每个业务域建立 Service 类（如 `WorkOrderService`、`EventService`），将数据访问封装为方法：

```cpp
// WorkOrderService.h
struct WorkOrderQuery { QString keyword; int statusFilter = -1; int page = 1; int pageSize = 20; };
struct WorkOrderListResult { QList<WoWorkOrder> items; int totalCount; };

class WorkOrderService {
public:
    static WorkOrderListResult queryWorkOrders(const WorkOrderQuery& params);
    static bool createWorkOrder(const WoWorkOrder& order);
    static bool updateStatus(qint64 id, int newStatus);
};
```

**3. 无限嵌套 Lambda — 可读性灾难**

页面逻辑大量使用嵌套 lambda，深度达到 4-5 层。例如 `loadEvents` 是一个 `std::function<void()>` 持有在 `shared_ptr` 中，内部又嵌套 action handler lambda、dialog lambda：

```cpp
auto loadEventsPtr = std::make_shared<std::function<void()>>();
*loadEventsPtr = [page, table, searchEdit, filterCombo, myTasksCheck, evActionHandlerPtr, loadEventsPtr, emptyHint, pb]()
{
    auto& loadEvents = *loadEventsPtr;  // 自引用 shared_ptr 循环
    // ... 200行逻辑 ...
};
```

这种 `shared_ptr<std::function>` 自引用模式是典型的 **代码沼泽**，不仅难以调试，还存在潜在的循环引用风险。

→ **重构方案**：将 lambda 提取为类的成员函数，用信号/槽替代回调嵌套。

**4. `using namespace UiKit` 全局污染**

[MainWindow.cpp:5](file:///d:/smart_community_management/src/widgets/MainWindow.cpp#L5)、每个 Page.cpp 文件头部均有 `using namespace UiKit;`，在 `.cpp` 文件顶层使用 `using namespace` 会导致命名空间污染，尤其是 `UiKit` 内有 `fadeInWidget`、`showToast` 等通用名称。

→ 改为 `using UiKit::createPageHeader;` 按需引入，或直接使用 `UiKit::` 前缀。

**5. `PagesCommon.h` — 巨型预编译头**

[PagesCommon.h](file:///d:/smart_community_management/src/PagesCommon.h) 引入了 **62 个** 头文件（包括所有 Qt Charts 模块），所有页面 `.cpp` 都包含它。这严重拖慢编译速度，且违反最小依赖原则。

→ **修复方向**：每个 `.cpp` 文件仅包含自己需要的头文件。将 Charts 相关引入移到 `ReportPage.cpp` 内部。

#### 改进建议（强烈建议）

**6. Models.h 中模型类无行为**

[Models.h](file:///d:/smart_community_management/src/models/Models.h) 包含 583 行纯数据结构体，没有任何方法（连 `isValid()`、`displayName()` 都没有）。所有 ORM 映射逻辑散落在各页面的 SQL 查询中。

→ 为每个 Model 添加 `fromQuery(QSqlQuery&)` 静态工厂方法和 `toVariantMap()` 序列化方法。

**7. Constants.h 枚举模式重复冗余**

[Constants.h](file:///d:/smart_community_management/src/models/Constants.h) 中 20+ 个命名空间，每个都有几乎一模一样的 `label(int v)` 和 `color(int v)` 函数。大量 magic number 在使用点（如 `status == 0`、`userType == 3`）。

→ 使用 `Q_ENUM` 或宏定义消除重复，或改为统一的 `EnumHelper<T>` 模板。

**8. 密码明文存储在数据库中**

[AIService.h:165](file:///d:/smart_community_management/src/services/AIService.h#L165):

```cpp
if (key == "ai_api_key") m_apiKey = val;
```

API Key 以明文存储在 SQLite `sys_config` 表中，任何能访问 `.db` 文件的人都可以提取。

→ 至少在存储时使用操作系统凭据管理器或简单的加密。

**9. 编号生成有并发冲突风险**

[Utils.h:60-65](file:///d:/smart_community_management/src/utils/Utils.h#L60-L65):

```cpp
inline QString generateOrderNo() {
    QString prefix = "WO";
    QString timestamp = QDateTime::currentDateTime().toString("yyyyMMddHHmmss");
    int rand = QRandomGenerator::global()->bounded(1000, 9999);
    return prefix + timestamp + QString::number(rand);
}
```

秒级时间戳 + 4位随机数，在并发场景下有碰撞概率。

→ 使用毫秒级时间戳 + 序列号，或在数据库层使用自增序列。

**10. 数据库路径硬编码为相对路径**

[main.cpp:59](file:///d:/smart_community_management/src/main.cpp#L59):

```cpp
db.initialize("smart_community.db")
```

数据库文件创建在**当前工作目录**，而非应用程序目录或用户数据目录，可能导致多实例冲突或数据丢失。

→ 使用 `QStandardPaths::writableLocation(QStandardPaths::AppDataLocation)` 确定数据库路径。

#### 重构方案

**问题概述**：整个项目是一个"大泥球"架构 — UI、业务逻辑、数据访问完全混杂在一起。

**重构手法**：分层架构 + 提取类

```
当前:  Page.cpp → SQL → UI构建 → 事件处理 全混在一起
目标:  Page(UI) → Service(业务) → Repository(数据) → Model
```

**优先级排序**：

1. 先提取 Service 层（不改 UI，只把 SQL 移到 Service）
2. 再将巨型 Page 文件拆为独立子类
3. 最后清理 PagesCommon.h、Constants.h

#### 亮点（值得肯定）

- [DatabaseManager](file:///d:/smart_community_management/src/database/DatabaseManager.h) 的表白名单校验（`isKnownTable`）和标识符校验（`isValidIdentifier`）是良好的 SQL 注入防御
- [Utils.h](file:///d:/smart_community_management/src/utils/Utils.h) 的密码哈希使用 PBKDF2-SHA512 + 随机盐，并保留旧格式兼容，工程化处理得当
- [BasePage](file:///d:/smart_community_management/src/pages/BasePage.h) 通过信号解耦页面与 MainWindow 的设计思路正确
- QSS 样式系统（[style.qss](file:///d:/smart_community_management/resources/style.qss)）有完整的设计令牌（Design Token）体系，颜色、间距、字号层次清晰

**1. 技术驱动 > 需求驱动**

→ **纠偏建议**：暂停 AI 功能迭代，优先补齐权限控制和安全加固。

**2. "课程实训项目"定位与实际复杂度不匹配**

登录页底部标注"课程实训项目"，但系统有 30+ 张数据表、20+ 个功能模块、完整的多角色设计。如果这真是实训项目，范围远超课程要求；如果是真实产品，则不应标注"实训"。

→ **影响**：定位模糊导致质量标准不清 — 用"实训"来降低质量期望是不合理的。

**3. 演示数据服务占据过大代码量**

[DemoDataService.cpp](file:///d:/smart_community_management/src/services/DemoDataService.cpp) 1618 行，占项目总代码量约 10%。演示数据是开发辅助工具，不应与业务逻辑同等优先级。

→ **建议**：将演示数据改为 SQL 脚本或 JSON 种子文件，减少 C++ 代码量。

### 🎨 视觉审查报告

#### 整体印象

- **一句话总评**：配色克制、层次清晰，是一个在 Qt 桌面框架下达到"及格线以上、专业线边缘"的视觉设计，但在细节打磨和信息架构上仍有明显提升空间。
- **视觉评分：6.5 / 10**

#### 严重视觉问题（让我想关掉页面）

**1. 侧边栏信息密度失衡**

侧边栏 240px 固定宽度，深蓝色背景（`#0f172a`），分组标题字号仅 11px，字色 `rgba(255,255,255,0.40)` — 几乎看不清。30+ 个菜单项挤在一个滚动区域中，手风琴模式虽然缓解了空间压力，但折叠/展开没有动画过渡，生硬切换让人迷失位置感。

**2. 登录页品牌区"建筑线条"过于简陋**

[LoginWidget.cpp:33-54](file:///d:/smart_community_management/src/widgets/LoginWidget.cpp#L33-L54) 中的 `BrandPanel` 使用 `drawLine` 和 `drawRoundedRect` 硬编码绘制"城市天际线"。这种几何线条建筑感的方向是对的，但实际效果像是一个 **debug 占位图** — 线条间距固定（40px 网格），矩形随机感不足，缺乏层次和景深。

→ 应替换为高质量的 SVG 插画或预渲染图片。

**3. 卡片阴影被完全禁用**

[UiKit.cpp:56-60](file:///d:/smart_community_management/src/ui_kit/UiKit.cpp#L56-L60):

```cpp
void applyCardShadow(QWidget* card) {
    card->setGraphicsEffect(nullptr);
}
```

所有调用 `applyCardShadow` 的地方实际上什么都没做！统计卡片、对话框、面板全部是 **纯平面 + 1px border**，缺乏立体感和层次感。

→ 使用 `QGraphicsDropShadowEffect` 或 QSS `box-shadow`（Qt6 已部分支持）恢复阴影效果。

#### 视觉改进清单

**4. 表格操作列视觉权重不足**

操作列按钮被限制在 `max-width: 80px`、`max-height: 28px`、`font-size: 12px`，在 44px 行高的表格中显得极其弱小。用户视觉焦点会被表格数据吸引而忽略操作入口。

→ 操作列按钮可使用更醒目的样式（如圆角胶囊按钮 + 轻微背景色）。

**5. 面包屑导航字号过小**

面包屑使用 `font-size: 13px`，在 56px 高的顶栏中几乎被淹没。分隔符 `>` 使用 `#cbd5e1` 灰色，对比度不足。

→ 面包屑当前页名称应使用 15px + `font-weight: 600`，增加视觉权重。

**6. 统计卡片缺少视觉差异化**

报告页面和仪表板的统计卡片全部采用相同的白底 + 1px border + 顶部色条设计。当 4-6 个卡片并排时，视觉上过于均质化，用户无法快速定位关键指标。

→ 关键指标卡片可使用浅色背景（如 `#fff7ed`）与普通指标区分。

**7. 淡入动画名存实亡**

[UiKit.cpp:279-281](file:///d:/smart_community_management/src/ui_kit/UiKit.cpp#L279-L281):

```cpp
void fadeInWidget(QWidget* widget) {
    if (widget) widget->show();
}
```

所谓的"页面切换淡入动画"实际上只是 `widget->show()` — **完全没有动画效果**。
