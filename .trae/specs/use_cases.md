# 智慧社区管理平台 - 用例设计文档

> **版本**: v1.0  
> **更新日期**: 2026-06-16  
> **维护说明**: 本文档按角色视角组织用例，每个用例标注涉及的系统模块和数据库表。用例编号格式 `UC-XX-YYY`（XX=角色编号，YYY=用例序号）。

---

## 一、角色体系总览

系统定义了 **6 大域、21 种角色**，覆盖社区治理的完整参与方。

| 域 | 角色 | 角色Key | 数据权限 | 菜单范围 |
|---|---|---|---|---|
| **居民域** | 居民户主 | resident_owner | 个人级 | 工作台、小区管理（报修/投诉/公告/停车/缴费）、社区服务、智能问答 |
| | 家庭成员 | resident_family | 个人级 | 同上 |
| | 租户 | resident_tenant | 个人级 | 同上（停车/缴费受限） |
| | 楼栋长 | building_leader | 楼栋级 | 工作台、居民档案（楼栋内）、小区管理 |
| | 志愿者 | volunteer | 个人级 | 工作台、志愿服务、社区服务 |
| **物业域** | 物业客服 | property_cs | 小区级 | 工作台、基础档案、报修/投诉/巡检/公告/停车/缴费、工单统计 |
| | 物业管家 | property_steward | 小区级 | 同客服 + 访客管理 |
| | 维修人员 | property_repair | 个人级 | 工作台、报事报修（仅自己负责的） |
| | 保安人员 | property_security | 小区级 | 工作台、访客管理、停车管理 |
| | 物业经理 | property_manager | 小区级 | 工作台、基础档案、小区管理全部、工单统计/综合看板 |
| **治理域** | 网格员 | grid_worker | 社区级 | 工作台、居民档案、网格管理、特殊群体、事件中心、巡查管理、重点关怀、事件统计 |
| | 社区工作人员 | community_worker | 社区级 | 工作台、基础档案、事件中心/巡查/重点关怀/督办/民意/志愿服务/便民/统计 |
| | 社区书记 | community_secretary | 社区级 | 同工作人员 + 考核管理 |
| | 街道工作人员 | street_worker | 街道级 | 工作台、事件中心、督办管理、考核管理、事件统计/综合看板 |
| | 街道领导 | street_leader | 街道级 | 同街道工作人员 |
| **服务域** | 便民服务商 | service_provider | 协同级 | 工作台、便民服务（仅自己的工单） |
| | 社会组织管理员 | social_org_admin | 协同级 | 工作台、志愿服务 |
| **监督域** | 业委会成员 | oc_member | 小区级 | 工作台、公告通知、业委会议题、公共收益、综合看板 |
| | 业委会主任 | oc_director | 小区级 | 同成员 + 物业缴费监督 |
| **平台域** | 超级管理员 | super_admin | 平台级 | 全部菜单 |
| | 安全审计管理员 | audit_admin | 平台级 | 系统管理（操作日志、登录日志） |
| | 数据分析管理员 | data_admin | 平台级 | 统计分析全部 |

## 二、用例设计（按角色分组）

### 2.1 居民户主

**涉及表**: `cm_resident`, `cm_house`, `cm_house_resident`, `wo_work_order`, `nt_announcement`, `cm_vehicle`, `pm_bill`, `cm_parking_space`, `sv_volunteer`, `sv_volunteer_activity`, `sv_volunteer_signup`, `ev_evaluation`, `ge_opinion`

#### UC-01-001: 居民报修
**场景**: 张三家厨房水管漏水，需要报修。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 登录系统，查看工作台 | 显示个人报修历史、待处理工单数 |
| 2 | 进入"报事报修"，点击"新建" | 弹出表单：选择类型（水电维修）、填写标题、描述、位置、上传照片 |
| 3 | 提交报修 | 生成工单号，状态=待受理(0)，插入 `wo_work_order`，记录 reporter_id |
| 4 | 查看工单详情 | 显示工单状态流转时间线 |
| 5 | 物业完成维修后，对工单评价 | 插入 `ev_evaluation`，工单状态变为已评价(6) |

**验证点**: 居民只能看到自己提交的工单（`WHERE reporter_id = 当前用户`）。

#### UC-01-002: 居民投诉物业
**场景**: 小区垃圾三天未清理，居民投诉物业。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"投诉建议"，新建投诉 | 选择类型、填写投诉内容 |
| 2 | 提交 | 插入 `wo_work_order`（order_type 对应投诉类型），状态=待受理 |
| 3 | 物业处理后，查看处理结果 | 工单状态变为已完成，显示处理说明 |
| 4 | 不满意处理结果，再次投诉 | 新建工单，关联原工单 |

#### UC-01-003: 查看公告通知
**场景**: 居民查看小区最新公告。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"公告通知" | 查询 `nt_announcement`，筛选 target_scope 包含当前用户所属小区的公告 |
| 2 | 点击查看公告详情 | 显示标题、内容、发布时间、发布人 |
| 3 | 标记已读 | 更新 `nt_notification.is_read` |

#### UC-01-004: 缴纳物业费
**场景**: 张三收到物业费账单，在线查看并缴费。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"物业缴费" | 查询 `pm_bill` WHERE house_id 关联当前用户的房屋 |
| 2 | 查看账单明细 | 显示费用类型、金额、账期、缴费状态 |
| 3 | 查看历史缴费记录 | 可按年份、类型筛选 |

#### UC-01-005: 管理车辆信息
**场景**: 张三新增一辆车，需要登记。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"车辆管理" | 显示已登记车辆列表 |
| 2 | 新增车辆 | 填写车牌号、品牌、颜色、类型，关联小区 |
| 3 | 绑定车位 | 选择空闲车位 `cm_parking_space`，状态变为已占用 |

#### UC-01-006: 报名志愿活动
**场景**: 张三看到社区组织的环保志愿活动，想参加。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"志愿服务" | 查看招募中的活动列表 |
| 2 | 点击"报名" | 插入 `sv_volunteer_signup`，活动 enrolled_count +1 |
| 3 | 活动当天签到 | 更新 checkin_time |
| 4 | 活动结束签退 | 更新 checkout_time，计算志愿服务时长 |

#### UC-01-007: 提交民意
**场景**: 居民对小区绿化改造提出建议。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"民意收集"（如菜单可见） | 查看已提交的民意 |
| 2 | 新建民意 | 选择类别、填写内容 |
| 3 | 查看处理进度 | 社区工作人员处理后更新状态 |

#### UC-01-008: AI 智能问答
**场景**: 居民想知道物业费缴纳标准。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"智能问答" | 显示聊天界面 |
| 2 | 输入"物业费怎么算" | 匹配 `ai_knowledge`，返回答案 |
| 3 | 评价回答是否有用 | 更新 `ai_chat_log.is_useful` |

### 2.4 物业客服

**涉及表**: `wo_work_order`, `cm_estate`, `cm_house`, `cm_resident`, `cm_vehicle`, `nt_announcement`, `pm_bill`, `ge_inspection`

#### UC-06-001: 受理报修工单
**场景**: 收到居民报修，客服需要受理并派单。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"报事报修" | 查看所有待受理工单 |
| 2 | 点击"受理" | 工单状态变为已受理(1)，记录 accept_by、accept_time |
| 3 | 分派给维修人员 | 工单状态变为已派单(2)，记录 assign_to、assign_time |
| 4 | 跟踪维修进度 | 查看状态变化：处理中(3)→已完成(4) |

#### UC-06-002: 处理投诉
**场景**: 收到居民投诉，需要调查并回复。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 查看待处理投诉 | 筛选 order_type 为投诉类的工单 |
| 2 | 受理投诉 | 状态变更 |
| 3 | 填写处理结果，标记完成 | 更新 result_desc、finish_time |
| 4 | 等待居民评价 | 工单进入已评价状态 |

#### UC-06-003: 发布公告
**场景**: 物业发布停水通知。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"公告通知"，新建 | 选择类型=物业公告，填写标题和内容 |
| 2 | 设置目标范围 | 选择"翠苑小区全体居民" |
| 3 | 发布 | 插入 `nt_announcement`，推送到居民消息中心 |

#### UC-06-004: 查看工单统计
**场景**: 物业客服查看本月工单处理情况。
| 步骤 | 操作 | 系统行为 |
|------|------|----------|
| 1 | 进入"工单统计" | 按类型/状态/时间维度展示饼图和柱状图 |
| 2 | 查看超时工单 | 筛选 SLA 超期的工单 |

#### UC-11-001: 日常巡查上报事件

#### UC-12-004: 督办管理

#### UC-12-005: 处理便民服务工单

| 模块 | 评估 |

| 督办管理 | **设计合理**：事件超时→督办→反馈，体现治理闭环 |
| 考核管理 | **设计合理**：指标配置→结果录入→排名，支撑社区绩效评估 |
| 民意收集 | **设计合理**：居民提交→分类→处理→反馈，体现民主参与 |
| 重点关怀 | **设计合理**：特殊群体→走访记录→跟进，体现人文关怀 |

### 5.3 不冗余的模块说明

每个模块在用例中都有明确的使用场景和角色覆盖：

- **基础档案**（8个子模块）：作为所有业务模块的数据底座，组织→小区→楼栋→房屋→居民→车辆→设施→网格→特殊群体，形成完整的社区数据链路
- **小区管理**（9个子模块）：覆盖物业日常运营的全部场景
- **社区治理**（6个子模块）：覆盖社区治理从事件发现到考核评价的全链路
- **社区服务**（3个子模块）：覆盖便民、志愿、就业三大服务方向
- **统计分析**（4个子模块）：从工单/事件/服务/综合四个维度提供数据支撑
- **系统管理**（6个子模块）：用户-角色-菜单-字典-日志-智能问答，系统运行的基础保障
