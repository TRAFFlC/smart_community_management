## 一、角色流程模拟

选取 5 个核心角色，模拟从登录到完成核心任务的完整操作路径，标注每一步对应的用例编号和页面模块（菜单 ID）。

**核心任务**: 报修 → 查看公告 → 缴费 → 报名志愿活动

### 4.1 现状

### 4.2 对平台体验的影响

#### 4.2.1 待办中心缺失的影响

**严重影响**，原因如下：

1. **工作台待办卡片不完整**：工作台（1）的"待办事项"卡片仅展示 3 项静态统计（待处理工单数/待审核事件数/近期公告数），**不可点击跳转**，用户无法快速定位到具体待办事项。

2. **跨模块待办无聚合**：当前各模块（报修/事件/督办/民意/便民）的待办分散在不同页面，用户需要逐个进入页面查看。对于物业客服、社区工作人员等需要处理多类待办的角色，效率极低。

3. **角色工作流断裂**：例如物业客服登录后，需要分别进入"报事报修"看待受理工单、"投诉建议"看待处理投诉、"公告通知"看待发布公告，无法在一个统一入口看到所有待办。

4. **影响 P0 功能闭环**：报修全流程、事件处理全流程等 P0 核心流程的"待受理→待处理→待评价"等环节，缺少统一待办入口会导致任务遗漏。

#### 4.2.2 消息中心缺失的影响

**严重影响**，原因如下：

1. **公告推送链路断裂**：用例 UC-01-003、UC-06-003 描述公告应"推送到居民消息中心"，但 `nt_notification` 表从未被写入，居民无法收到公告通知。当前居民只能主动进入"公告通知"页面查看，违背了"推送"设计意图。

2. **工单状态变更通知缺失**：用例 UC-01-001 步骤4 描述"通知完成"，但代码中工单状态变更后未插入 `nt_notification`，居民无法及时知道工单进度。

3. **跨角色协作通知缺失**：报修派单后维修人员无通知、事件分派后处理人无通知、督办创建后被督办人无通知、志愿活动报名后社区无通知……所有跨角色协作均依赖用户主动查看，效率极低。

4. **顶部通知铃铛无响应**：`MainWindow.cpp` 第 292-298 行实现了通知铃铛按钮，但**未连接任何点击事件**，点击无反应，是"无响应按钮"（违反项目规则第 6 条）。

5. **月卡到期提醒缺失**：用例 UC-09-003 步骤5 描述"月卡即将到期系统提前7天推送提醒"，但代码中无此逻辑。

### 4.3 是否应该优先实现

**强烈建议优先实现**，理由如下：

| 维度 | 待办中心 | 消息中心 |
|------|---------|---------|
| 影响范围 | 全部角色（21 种） | 全部角色（21 种） |
| 影响频率 | 每日高频 | 每日高频 |
| 影响程度 | 🔴 严重影响 P0 功能闭环 | 🔴 严重影响跨角色协作 |
| 实现难度 | 🟡 中（聚合查询各模块待办） | 🟡 中（读写 `nt_notification` 表） |
| 数据基础 | ✅ 各业务表已存在 | ✅ `nt_notification` 表已存在 |
| 优先级建议 | **P0**（应立即实现） | **P0**（应立即实现） |

#### 4.3.1 待办中心实现建议

1. **统一待办列表**：聚合查询各模块待办事项
   - 工单待受理：`SELECT * FROM wo_work_order WHERE status = 0 AND del_flag = 0`
   - 事件待审核：`SELECT * FROM ge_event WHERE status = 0 AND del_flag = 0`
   - 督办待反馈：`SELECT * FROM ge_supervision WHERE status = 0 AND del_flag = 0`
   - 民意待处理：`SELECT * FROM ge_opinion WHERE status = 0 AND del_flag = 0`
   - 便民待接单：`SELECT * FROM sv_service_order WHERE status = 0 AND del_flag = 0`
2. **按角色过滤**：根据当前用户角色显示对应待办（如维修人员只看分派给自己的工单）
3. **快捷跳转**：点击待办项跳转到对应模块详情页
4. **待办统计**：按类型/优先级统计待办数量

#### 4.3.2 消息中心实现建议

1. **通知列表**：查询 `nt_notification` 表，按时间倒序展示
2. **通知类型**：公告推送、工单状态变更、事件分派、督办创建、志愿活动报名、月卡到期提醒等
3. **已读/未读**：支持标记已读、批量已读、未读数量统计
4. **通知触发点**：在各业务模块的状态变更处插入 `nt_notification` 记录
5. **顶部铃铛联动**：将通知铃铛连接到消息中心页面，显示未读数量角标

## 五、总结

### 5.1 整体评估

| 维度 | 评估 |
|------|------|
| 用例覆盖度 | ✅ 优秀：21 种角色、40+ 用例、6 大域全覆盖 |
| 数据建模 | ✅ 优秀：表结构完整，命名规范，审计字段齐全 |
| 查询展示 | ✅ 良好：所有页面均从数据库查询，无硬编码数据，搜索/筛选控件均已连接 |
| 写入操作 | ❌ 严重不足：仅 4 处 insert（工单/投诉/公告/事件）+ 1 处 insert（志愿报名），大量新建/编辑/删除操作缺失 |
| 状态流转 | ❌ 严重不足：工单仅支持"受理"和"完成"两步，事件仅支持"审核"和"完成"两步，缺少中间状态 |
| 跨角色协作 | ❌ 严重不足：派单、分派、督办、通知等协作环节几乎全部缺失 |
| 数据权限 | ❌ 缺失：所有查询无 `WHERE reporter_id = 当前用户` 过滤，所有用户看到全量数据 |

### 5.2 优先修复建议

按影响程度排序，建议按以下顺序修复：

1. **P0 - 数据权限过滤**：所有查询添加 `WHERE reporter_id = 当前用户` 或按角色数据权限过滤（影响安全）
2. **P0 - 待办中心 + 消息中心**：实现统一待办和通知，打通跨角色协作链路（影响体验）
3. **P0 - 工单派单 + 评价**：补全报修全流程的派单和评价环节（影响核心闭环）
4. **P0 - 事件分派 + 升级督办**：补全事件处理全流程的分派和督办环节（影响核心闭环）
5. **P1 - 各模块新建操作**：补全访客登记、志愿活动创建、议题发起、督办创建、民意处理、便民派单、岗位发布等新建操作（影响功能完整性）
6. **P1 - 巡查/走访/巡检执行**：补全网格员巡查、走访特殊群体、物业巡检的执行入口（影响日常作业）
7. **P1 - 公告推送 + 已读标记**：实现公告推送到消息中心，居民标记已读（影响信息触达）
8. **P2 - 缴费支付 + 月卡办理**：实现演示级支付流程和月卡办理（影响业务闭环）
9. **P2 - 统计分析增强**：工单/事件/服务统计增加统计卡片、趋势折线图、数据表格（影响数据洞察）
10. **P2 - 导出功能**：实现统计报表导出（影响数据分析）
