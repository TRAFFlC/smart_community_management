# 智慧社区管理平台任务追踪列表

## 1. 文档目标

本文档用于将《智慧社区管理平台》的系统规格转化为可执行任务，适配 6 人团队、30 天开发周期，便于任务分工、每日跟踪、里程碑检查和答辩复盘。

## 2. 团队分工建议

建议采用“角色分工明确，但关键模块双人联动”的方式：

- 成员 A：项目负责人 / 产品经理 / 统筹文档。
- 成员 B：前端负责人，负责管理端框架与核心页面。
- 成员 C：前端开发，负责居民端与通用组件。
- 成员 D：后端负责人，负责基础框架、权限、数据库设计。
- 成员 E：后端开发，负责核心业务模块。
- 成员 F：测试 / 运维 / UI 支持 / 数据与答辩材料整理。

## 3. 总体里程碑

### M1：需求与架构冻结（第 1-5 天）

- 完成角色与业务边界确认。
- 完成系统规格文档。
- 完成任务分工与排期。
- 完成原型草图和数据库草图。

### M2：基础框架完成（第 6-12 天）

- 完成前后端项目初始化。
- 完成登录、权限、组织、用户基础能力。
- 完成核心档案数据模型。

### M3：核心业务闭环完成（第 13-22 天）

- 完成报修工单闭环。
- 完成社区事件闭环。
- 完成公告通知和基础统计。
- 完成居民端基础访问能力。

### M4：联调优化与验收（第 23-30 天）

- 完成联调测试与缺陷修复。
- 完成演示数据、演示脚本和答辩材料。
- 完成文档归档与最终验收。

## 9. 风险追踪项

- 角色过多导致页面范围失控。

### 10.5 风险追踪补充

- 表结构过多（45+张），可能导致开发时频繁修改数据库，建议前期用 migration 工具管理版本。
- 演示数据量大，建议提前编写 SQL 种子脚本或 Java 数据初始化器。
- 多角色切换演示需要充分的测试账号（建议至少15个不同角色账号）。
- 智能机器人如果时间不足，可用纯前端模拟+关键词匹配代替AI接入。

## 11. 任务执行原则

- 优先做能形成闭环的模块，不优先堆砌零散页面。
- 优先做可答辩、可演示、可量化的功能。
- 高风险模块尽早实现最小版本。
- 每个任务必须明确负责人、交付物、依赖与截止时间。
- 每完成一个阶段，必须更新文档和演示说明。
